# WORKFLOW — как устроена работа над Stigla

Для знаний продуктового чата (Claude.ai Project) и `docs/` репозитория.
Отвечает на вопросы вида «мне сейчас ветку создавать или чат?».

## Роли

- **Продуктовый чат (этот Project)** — стратегия, бэклог, спеки, написание
  промптов-задач. Выход любого разговора — файл, который уезжает в `docs/` репо.
- **Claude Code** — исполнение. Сессия получает промпт-файл, работает в своей
  ветке-папке, пишет отчёт в `docs/reports/`, коммитит в свою ветку.
- **Репозиторий — единственный источник истины.** Чаты одноразовые: любой можно
  удалить без потерь, если всё важное доехало до файлов. Копии BACKLOG/WORKFLOW
  в знаниях проекта — слепки; мастер — в репо.

## Цикл задачи

```
идея → обсуждение в продуктовом чате → промпт-файл → docs/prompts/
  → новая сессия Code в worktree ветки → работа → отчёт в docs/reports/ + коммиты
  → обновить design/gallery при визуальных изменениях
  → владелец читает отчёт (сначала блок безопасности, если есть)
  → «влить в main» → merge + релиз при необходимости
  → отчёт приносится в продуктовый чат → обновление BACKLOG.md
  → прибрать worktree и ветку
```

## Дерево решений

**Новая крупная задача (фича, большой рефакторинг)**
→ Новая ветка + worktree + новый чат Code, подключённый к папке worktree:
```sh
git worktree add ../stigla-<имя> -b feature/<имя>
cd ../stigla-<имя> && flutter pub get   # и/или npm install в backend/
```
Промпт задачи — файлом в `docs/prompts/` (он публичный: без секретов и инфра-деталей).

**Продолжение той же задачи** → тот же чат Code.
Чат распух и начал тупить → закрыть, открыть новый чат на ту же папку:
контекст поднимется из CLAUDE.md, спеки и отчётов, а не из памяти чата.

**Мелкий баг/правка** → ветка `fix/<имя>`. Отдельный worktree не обязателен,
если параллельно в main-папке ничего не запущено; после фикса — быстрый merge.

**Вопрос «что делать дальше» / продуктовое обсуждение** → продуктовый чат
(внутри Project, чтобы история копилась там).

**Приёмка задачи** → читать отчёт в `docs/reports/`; если в промпте был чек-лист —
сверять по нему. Если задача меняла UI — сверять, что слепки в `design/gallery/`
обновлены (новые/изменённые состояния, `index.html`) и `preview-gallery`
передеплоен. Найденные секреты ротировать до любого пуша. Merge в main —
только явной командой.

**Релиз** → только из main:
```sh
cd backend && npm run deploy
cd ../app && flutter build web --release --dart-define-from-file=dart_defines.json
npx wrangler pages deploy build/web --project-name=stigla --branch=main
```
Проверять деплой по sha `main.dart.js` (curl), не по браузеру — кастомный домен
кэширует (см. CLAUDE.md, gotchas).

**После merge** → прибрать:
```sh
git worktree remove ../stigla-<имя>
git branch -d feature/<имя>
```
и обновить `docs/BACKLOG.md` (статусы), при вехах — обновить слепки в Project
(промпт `docs/prompts/prompt_handoff_package.md` собирает папку `handoff/`).

## Staging: ветка = своя пара превью

Общего staging-слота больше нет. Ветка, которой нужен живой стенд, получает:

- **бэкенд** — версию воркера: `npm run staging:version` (обёртка над
  `wrangler versions upload`) печатает preview-URL версии. Активный деплой
  staging-воркера при этом не меняется — версии веток живут параллельно.
- **фронт** — Pages-превью, собранное с `API_BASE_URL=<preview-url версии>`:
  `wrangler pages deploy ... --branch=preview-<имя>`.

Детали и ограничения — `docs/staging.md`. Главные:
- Секреты и биндинги версия наследует от воркера; **D1 и KV общие для всех
  версий**. Ветка, меняющая схему D1 или семантику KV-флагов, — [STOP & ASK]
  в промпте или своя копия базы.
- Cron-триггеры на preview-версиях не выполняются.

## Правила гигиены

- `main` всегда релизопригоден; недоделанное в main — только за фиче-флагом.
- Одна сессия Code = одна ветка = один worktree. Две задачи в одной папке — конфликт.
- Репо публичный (AGPL-3.0): секреты только в `.dev.vars` / `dart_defines.json`
  (gitignored) и Cloudflare secrets; промпты и отчёты пишутся как публичные тексты.
- Не выжимать из сессии Code недели работы: задача сдана → отчёт → новая сессия.
- Ветки не деплоят в общий staging-воркер (`wrangler deploy` со staging-конфигом
  из ветки — запрещено). Только `versions upload` + Pages-превью.
- Код/комментарии — EN; отчёты, доки, коммуникация — RU.

## Быстрая шпаргалка

```sh
git worktree list        # что где
git status               # изменения в текущей папке
cd ~/Projects/stigla && git checkout main && git pull   # перед merge
git merge feature/<имя> && git push                      # влить готовое
```
