# Карта покрытия транспортом (Coverage Map)

Спека фичи. Ветка: `feature/coverage-map`. Промпт задачи: `docs/prompts/prompt_coverage_map.md`.

## Зачем

Отдельная вкладка-инфографика: светящиеся коридоры маршрутов поверх тёмной
карты (референсы: Citi Bike trips map, Strava heatmap). Отвечает на вопрос
«куда вообще дотягивается транспорт и где его много». Играбельная, скриншотная,
шерибельная — организический крючок в связке со screenshot badge.

Это **не навигация**: слой не участвует в основном сценарии «что едет к моей
остановке», живёт на отдельном экране.

## Визуальный язык

V0 — «Citi Bike-стиль»: дискретные линии по геометрии маршрутов,
data-driven ширина и яркость по весу сегмента. Строго line layer,
без растрового blur/glow (Strava-стиль — возможная полировка позже,
отдельной задачей).

- Подложка — та же, что в основной карте: `core/map_style.dart`,
  theme-synced (светлая/тёмная). Отдельный стиль карты НЕ заводим.
- На тёмной теме линии — тёплая шкала (оранжевый → белый по росту веса),
  на светлой — подобрать читаемую шкалу (например, синий → тёмно-синий);
  точные цвета — на усмотрение исполнителя, критерий: сегменты с высоким
  весом очевидно ярче/толще.
- Цвет по типу ТС в режиме фильтра: трамвай / троллейбус / автобус —
  переиспользовать классификацию и цветовые ассоциации из `map_support.dart`.

## Данные и вес

Источник геометрии — GTFS shapes (`public/gtfs/`, уже раздаются статикой).
Бэкенд-изменений в V0 нет.

Вес сегмента V0 = **число маршрутов, проходящих по сегменту**. Алгоритм:

1. Взять все shapes всех маршрутов.
2. Разбить на сегменты, совпадающие сегменты разных маршрутов схлопнуть
   (совпадение — по геометрии с допуском; точный метод — на усмотрение
   исполнителя, но результат должен быть предвычислен, не на лету в кадре).
3. Каждому сегменту — count маршрутов и список типов ТС.
4. Результат — предвычисленный GeoJSON, собираемый скриптом на этапе
   `gtfs:build` (расширить `scripts/build-gtfs.mjs` или добавить соседний
   скрипт), раздаётся как статика рядом с остальным GTFS.

V1 (не в этой задаче): вес = частота из GTFS-расписания; V2: вес = фактическая
интенсивность из агрегатов «линия × час×день» + фильтр по часу дня.
Структуру GeoJSON заложить так, чтобы добавление альтернативных весов
не ломало клиент (properties с запасом: `routes_count`, `types`, задел под
`frequency`, `observed_intensity`).

## UI

- Третий экран в `IndexedStack` (карта / идеи / покрытие), пункт в drawer.
- Контролы V0: фильтр по типу ТС (чипы: все / трамвай / троллейбус / автобус,
  мультивыбор). Заложить место под будущий контрол «час дня» (не реализовывать).
- Легенда: шкала веса (по образцу рефа Citi Bike — диапазоны count).
- Камера по умолчанию — весь Белград; позиция/зум независимы от основной карты.
- Фиче-флаг `coverage_map_show` (KV, по образцу `analytics_show`):
  OFF на проде, ON на staging.

## Производительность

- GeoJSON грузится один раз, слой статичный — никакого поллинга.
- Если предвычисленный файл получается тяжёлым (> ~1–2 МБ gzip) — упрощение
  геометрии (simplify) на этапе сборки, не на клиенте.
- Фильтр по типу — через setFilter/expression на слое, без пересборки источника.

## DoD

- На вкладке видна карта покрытия: густые коридоры очевидно ярче одиночных линий.
- Фильтр по типу работает, тема переключается вместе с остальным приложением.
- Файл покрытия собирается воспроизводимо из GTFS одним скриптом.
- За фиче-флагом, main релизопригоден.
