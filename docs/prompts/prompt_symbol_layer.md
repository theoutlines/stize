# Задача: символьный GPU-слой для движущихся объектов (масштабируемый рендер ТС)

Ветка: `feature/symbol-layer` · Worktree: `../stigla-symbol-layer`
Отчёт: `docs/reports/2026-07-XX-symbol-layer.md` (RU)

## Зачем (архитектурное решение, не заплатка)

Текущий рендер ТС — Flutter-виджеты (`WidgetLayer`/`Marker`) поверх карты. Замеры
доказали: стоимость **линейна по числу объектов** (мало ТС → 60 fps; ~40 → 19 fps),
плюс каждый объект проецируется geo→screen. Это не масштабируется под роадмап:
метро, электрички, микромобильность (самокаты/велосипеды), и особенно официальный
API со ВСЕМ городом (сотни объектов в кадре вместо текущего fan-out ≤12 остановок).

Символьный слой карты (MapLibre `SymbolLayer` по GeoJSON-источнику) рисует объекты
батчем на GPU — стоимость **сублинейна** по количеству (40 или 400 символов ≈ одна
цена). Так делают все зрелые карты движущихся объектов (Яндекс, Flightradar24,
MarineTraffic — наши же бенчмарки). Это покупает потолок на годы, а не на спринт.

**Важно — цена перехода в нашем случае низкая:** наша пилюля несёт ровно то же,
что символы Яндекса/Belgrade Bus (номер + тип + стрелка направления) — богатства
на маркере нет. Fleet-ID и вся идентичность живут в шторке по тапу, НЕ на маркере.
Значит символьный слой ничего из показываемого на карте не теряет.

Код/комментарии EN, отчёт RU, публичный текст без секретов/upstream-деталей.

## Проектировать под роадмап, а не только под автобусы

Это ключевое требование, чтобы не переделывать через полгода:
- Слой — **«движущиеся объекты с типизацией»**, не «слой автобусов». Заложить
  модель, где у объекта есть тип (bus/tram/trolley, а в будущем metro/train/
  scooter/bike), и рендер (иконка/цвет/поведение) выбирается по типу data-driven
  выражениями MapLibre. Добавление нового типа = новая иконка + правило, без
  переписывания слоя.
- Источник данных абстрагировать так, чтобы будущие сущности из ДРУГИХ источников
  (не GSP) могли влиться в тот же слой.

## Что переиспользовать (из донора `feature/timed-trajectory`)

Ознакомься с веткой `feature/timed-trajectory` (не мержим, донор):
- **timed-математика A′** (план движения вперёд по `second_left_by_route` +
  геометрия маршрута) — переиспользовать ЦЕЛИКОМ. Меняется только приёмник: вместо
  позиции виджета писать координаты в **GeoJSON-источник** символьного слоя
  (`setData`/feature update), карта анимирует символы сама.
- fps-оверлей за флагом — забрать как инструмент замера.
- Аффинная проекция из timed-trajectory для символьного слоя НЕ нужна (карта сама
  проецирует символы) — не тащить, но знать, что она там есть.

## Реализация

### Рендер
- ТС на карте → `SymbolLayer` по GeoJSON-источнику. Символ несёт: иконку типа
  (bus/tram/trolley — цвета из `map_support.dart`), текст номера линии
  (`text-field`, фон через `icon-text-fit` для «пилюли»), поворот стрелки
  направления по heading (`icon-rotate`, heading уже считается на бэкенде).
- **Fleet-ID / идентичность — НЕ на символе.** Тап → та же bottom-sheet, что
  сейчас. Правило архитектуры: маркер на карте тупой и быстрый, богатство в шторке.

### Анимация
- Позиции ТС двигаются timed-математикой (донор), записываются в GeoJSON-источник;
  карта рендерит на GPU. Плавность — за счёт частого обновления координат в
  источнике + GPU-интерполяции, без виджет-repaint.
- **idle=0 сохранить:** когда объекты стоят (план исчерпан) — не гнать обновления
  источника впустую; в фоне/скрытой вкладке — пауза (как в термо-фиксе).

### Взаимодействие
- Тап по ТС → `queryRenderedFeatures` по символьному слою → открыть шторку линии
  (существующую). Переписать с виджет-хендлера.
- **Spiderfy** совпадающих ТС и **подсветка выбранного** — на символьном слое
  через feature-state / раздвижку координат в источнике (не виджетная геометрия).
  Сохранить поведение F4 (совпадающие ТС читаемо разведены).
- **Аварийный люк (опционально):** для ОДНОГО выбранного ТС, если нужен богатый
  виджет (подсветка маршрута) — это один объект, дёшево, допустимо оставить
  виджетом поверх. Масса — символы. Не обязательно в этой задаче.

### Безопасность внедрения
- **За фиче-флагом** (напр. `symbol_layer`): OFF на проде, ON на staging. При OFF —
  СТАРЫЙ виджетный путь работает как сейчас (не удалять его в этой задаче).
  Старый и новый пути живут параллельно, переключаются флагом. Прод не рискует.
- Деградация: не построился символьный слой → фолбэк на старый путь или пустой
  слой без краха.

## Definition of Done (CLAUDE.md)
1. Код в ветке; тесты зелёные. Юнит-тесты на модель типизированных объектов и на
   запись позиций в GeoJSON-источник (чистая логика отдельно от карты).
2. Отчёт RU: архитектура слоя, что переехало из донора, замеры fps до/после при
   РАЗНОМ числе объектов (10 / 40 / искусственно 200+ если можно засимулировать) —
   показать сублинейность. Device-замер за владельцем через оверлей.
3. BACKLOG/STATUS: symbol-layer как основной путь рендера ТС; donor-цепочка
   зафиксирована; старый виджетный путь помечен «за флагом, удаляется после
   приёмки symbol-layer».
4. Галерея: маркеры/пилюли ТС изменились визуально → обновить соответствующие
   страницы `design/gallery/` (map-default и пр.) и передеплой preview-gallery.
5. «Как проверить» без терминала: сессия деплоит staging-превью сама; владельцу —
   URL + `?fps=1`, что смотреть: плавность при МНОГО ТС (центр), fps в бюджете,
   ТС сидят на улицах, тап открывает шторку, spiderfy/подсветка работают.
6. Merge в main — по явной команде владельца.

## Границы
- НЕ трогать: политику 30s-кэша, upstream, SWR, схему D1/KV.
- НЕ удалять старый виджетный путь в этой задаче (он фолбэк за флагом).
- Аудит данных линий (700 vs 71) — ОТДЕЛЬНАЯ ветка, не смешивать (она в бэкенде/
  маппинге, этот слой — рендер; конфликтов быть не должно, но не трогать её файлы).
