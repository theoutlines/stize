# Задача: аудит фиче-флагов — инвентаризация и сверка (прод vs staging vs код)

Ветка: `fix/flags-audit` (новая, от свежего main) · Отчёт:
`docs/reports/2026-07-14-flags-audit.md` (RU)

## Контекст

За сегодня в прод уехал большой массив, флаги ставились в разных сессиях напрямую
через KV (`--remote`), часто с оговорками «временно / откат наготове». Накопилось
достаточно флагов, чтобы состояние могло рассинхрониться:
- staging-флаг остался ON там, где в проде должен быть OFF (или наоборот);
- флаг введён «временно» и забыт;
- флаг есть в коде, но в KV не заведён → берётся дефолт (возможно не тот);
- рассинхрон staging vs прод (на staging ON для теста, в прод забыли выставить).

Задача — **инвентаризировать все флаги, свести их фактическое состояние (код-дефолт
/ прод-KV / staging-KV) в одну таблицу, найти рассинхроны и подозрительные, и
предложить приведение в порядок.** Правки состояния флагов — по решению владельца,
не автономно.

## Что сделать (аудит, НЕ менять состояние без команды)

### 1. Инвентарь флагов из кода
Собери полный список фиче-флагов из `backend/src/lib/featureFlags.ts` (и где ещё
объявляются). Для каждого: имя, дефолт в коде, что гейтит (какая фича/экран),
env-aware ли (разный дефолт staging/прод).

### 2. Фактическое состояние в KV — прод и staging
Для каждого флага прочитай реальное значение из:
- **прод-KV** (namespace прод, `--remote`);
- **staging-KV** (namespace staging, `--remote`).
Отдельно отметь флаги, которых в KV **нет** (тогда действует код-дефолт).
Так же для параметров-конфигов (напр. `config:nearby_schedule_stops`) — прод vs
staging значение.

### 3. Сводная таблица + анализ
Собери таблицу: `флаг | что гейтит | код-дефолт | прод-KV | staging-KV | ожидаемо | статус`.
Для каждого определи **ожидаемое** состояние по продуктовой логике:
- готовые фичи в проде → ON прод;
- сырые/черновые (напр. `analytics_show`) → OFF прод, ON staging;
- экспериментальные за флагом → OFF прод.
Помечай **рассинхроны и подозрительные**:
- прод-KV ≠ ожидаемому (сырое торчит в проде ИЛИ готовое скрыто);
- staging ≠ прод там, где должно совпадать;
- флаг в коде без записи в KV (действует дефолт — тот ли?);
- флаги-«сироты»: в коде есть, нигде не используются (мёртвый флаг) — кандидат на
  удаление;
- используется в коде, но не объявлен в списке флагов.

### 4. Известный контекст (сверить, не сломать)
Сегодня в проде ДОЛЖНЫ быть ON: `symbol_layer`, `timed_trajectory`,
`schedule_fallback` (список+карта), `nearby_list`, `nearby_sort_board`.
Должны быть OFF в проде (сырое/эксперимент): `analytics_show`,
`coverage_map_show`, `coverage_on_main_map` (если не влито). `analytics_collect` —
ON прод (сбор в фоне). Сверь факт с этим; расхождение — в статус «⚠ проверить».

## Границы
- **Состояние флагов НЕ меняй** — только читай и своди. Изменения (выставить
  недостающие, поправить рассинхрон, удалить мёртвые) — отдельным списком
  «предлагаю изменить», по команде владельца.
- Секреты/URL источника не светить (флаги — публичные имена, значения 0/1).
- Код EN, отчёт RU.

## DoD
1. Полная сводная таблица: код-дефолт / прод-KV / staging-KV / ожидаемо / статус
   по каждому флагу и конфиг-параметру.
2. Явный список **рассинхронов и подозрительных** (что не так, почему, что
   предлагается) — отдельной секцией, приоритизированный (сначала «сырое в проде»).
3. Список **мёртвых флагов** (в коде, нигде не гейтят) — кандидаты на удаление.
4. Список **недостающих в KV** (действует дефолт — подтвердить, что дефолт верный).
5. Отчёт RU + «Как проверить» (владельцу — таблица + что решить).
6. Изменения состояния — НЕ выполнены, вынесены в «предлагаю» под команду владельца.
