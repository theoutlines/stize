# Задача: staging-бэкенд без общего слота — Worker versions + preview URLs

## Контекст

Сейчас staging-воркер один на всех: каждая ветка делает `wrangler deploy` в него
и перезаписывает чужую работу (за последнюю неделю ветки дважды перетёрли друг
друга). Фронтовая половина проблемы уже решена — Cloudflare Pages даёт превью на
ветку (`<alias>.stigla.pages.dev`). Нужно то же самое для бэкенда.

Механизм: **Worker versions**. `wrangler versions upload` загружает версию
воркера, не меняя активный деплой, и возвращает уникальный preview-URL
(`<version-prefix>-<worker>.<subdomain>.workers.dev`). Версии разных веток живут
параллельно; секреты и биндинги (D1, KV, ASSETS) наследуются от воркера.
Известные ограничения preview-версий: cron-триггеры не выполняются; нужен
включённый workers.dev subdomain у аккаунта.

Цель: каждая ветка получает изолированную пару «Pages-превью + preview-версия
воркера», общий staging-слот больше никем не перезаписывается.

## Объём работ

1. **Проверить предпосылки.** Убедиться, что у staging-воркера включён
   workers.dev subdomain и `versions upload` действительно возвращает рабочий
   preview-URL с унаследованными секретами и биндингами. Если что-то из этого
   не так (subdomain выключен, биндинги не наследуются, парольная защита
   превью ломается на version-URL) — [STOP & ASK] с описанием, что именно
   упёрлось и какие есть варианты.
2. **Скрипт/команда для сессий.** Добавить в `backend/package.json` команду
   `npm run staging:version` (обёртка над `wrangler versions upload`),
   которая печатает preview-URL версии. Если wrangler отдаёт URL только в
   человекочитаемом виде — распарсить и вывести явно, чтобы следующий шаг
   (сборка фронта) мог его взять.
3. **Связка с Pages-превью.** Задокументировать (и по возможности обернуть в
   команду) сценарий: собрать web-бандл с
   `--dart-define=API_BASE_URL=<preview-url версии>` и выкатить
   `wrangler pages deploy ... --branch=preview-<имя>`. CORS на воркере должен
   пускать origin Pages-превью — проверить и поправить, если сейчас список
   origins жёсткий.
4. **Обновить `docs/staging.md`:**
   - новый порядок: staging-слот больше не деплоят ветки; ветка = версия
     воркера + Pages-превью;
   - как получить preview-URL версии и куда его подставить;
   - что общее между версиями (D1, KV, секреты), а что изолировано (код);
   - правило про общий стейт: ветка, меняющая схему D1 или семантику
     KV-флагов, обязана остановиться и согласовать это отдельно
     ([STOP & ASK] в промпте такой ветки), либо работать на своей копии базы;
   - ограничение: cron на preview-версиях не срабатывает.
5. **Тесты/проверка.** Выкатить версию из этой ветки, собрать Pages-превью,
   убедиться, что панель ходит в preview-версию (проверить по заголовкам/логам,
   не по ощущениям), и что активный staging-деплой при этом не изменился.

## Не делать

- Не трогать прод-воркер и прод-процесс релиза (WORKFLOW, раздел «Релиз»).
- Не заводить отдельные именованные воркеры на ветку — осознанно отвергнуто
  (дублирование секретов, мусор после merge).
- Не менять схему D1/KV.

## Definition of done

- Код и скрипты закоммичены в ветку задачи, тесты проходят.
- `docs/staging.md` актуален; изменения процесса отражены.
- Отчёт RU в `docs/reports/YYYY-MM-DD-staging-worker-versions.md`, в конце —
  раздел «Как проверить» для владельца без терминала: URL Pages-превью этой
  ветки, как убедиться, что она ходит в свою версию воркера, и как убедиться,
  что общий staging-слот не тронут.
- Merge в main — только по явной команде владельца.

Если по ходу выяснится, что versions upload не совместим с текущей парольной
защитой превью или SWR-кэшем (общий `caches.default` между версиями) —
[STOP & ASK], не изобретать обходные пути молча.
