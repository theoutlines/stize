# Задача: экран «Рядом» — nearby-список в духе Transit

Ветка: `feature/nearby-list` · worktree: `../stigla-nearby-list`

## Контекст

У Transit главный сценарий — ноль ввода: открыл приложение → GPS → список
всех линий, проходящих через остановки рядом, с ближайшими отправлениями.
Не карта и не поиск маршрута, а «я вышел на улицу — что мне светит».

У Stigla сейчас главный экран — карта. Задача: собрать за фиче-флагом
прототип nearby-списка и проверить, насколько формат ложится на наши данные.
Это эксперимент, не редизайн: карта остаётся главным экраном.

Бенчмарк-референс: `docs/BENCHMARKS.md`, раздел Transit.

## Что строим (v1, сознательно узкий)

Экран «Рядом»:

1. Берём геопозицию пользователя (пермишен уже есть у карты — переиспользовать
   существующий механизм, не плодить второй запрос доступа).
2. Находим остановки в радиусе до ~500 м (по GTFS-данным, которые уже есть).
3. Собираем прибытия по этим остановкам и группируем **по линии + направлению**
   (направление = конечная). Одна группа = одна строка списка.
4. Строка: номер линии (с типом — трамвай/троллейбус/автобус, классификация
   уже есть), конечная, имя ближайшей остановки этой линии, ETA ближайших
   1–2 отправлений. Если по линии есть данные Fleet-ID в этой ветке — НЕ
   интегрировать, это отдельная фича; строка без бейджей машин.
5. Сортировка v1 — по ближайшему ETA. Learning-сортировка (по частоте
   использования), пины избранного — вне скоупа, в отчёте отметить как «дальше».
6. Дедупликация: линия обслуживает несколько остановок в радиусе → показываем
   только ближайшую к пользователю остановку.
7. Pull-to-refresh + автообновление раз в 30 с (не чаще — жёсткое ограничение,
   см. CLAUDE.md про кеш).

## Бэкенд

Апстрим отдаёт прибытия только по одной остановке. Для nearby-списка нужен
агрегирующий эндпоинт, например `GET /api/v1/arrivals/nearby?lat=&lon=&radius=`:

- Реализация — по образцу `src/lib/vehicles.ts` (vehicles-in-area): тот же
  ограниченный fan-out (≤12 остановок, ≤1500 м, riding the shared 30s cache).
  **Не поднимать лимиты и не обходить SWR-кеш.**
- Ответ: массив групп «линия + направление» с прибытиями и ближайшей остановкой.
- Не забыть явный Hono-роут для CORS (gotcha из CLAUDE.md).
- [STOP & ASK] Если по ходу окажется, что разумнее переиспользовать
  существующий `/vehicles/nearby` и агрегировать на клиенте — остановись и
  опиши trade-off (лишний трафик vs новый эндпоинт), прежде чем выбирать.

## Клиент

- Новый экран в `presentation/`, данные через существующий слой
  (`data/` → `domain/` → провайдеры Riverpod). `AsyncValue.valueOrNull`,
  не `.value` (gotcha).
- Вход на экран: пункт в drawer. [STOP & ASK] Если по архитектуре напрашивается
  таб/IndexedStack-слот вместо drawer — опиши варианты, не решай сам.
- Фиче-флаг `nearby_list`: OFF на проде, ON на staging. Флаг выключен —
  пункта в drawer нет.
- Состояния: нет геопозиции (подсказка включить), пустой радиус (честное
  «рядом остановок нет»), офлайн (существующий паттерн деградации).
- Локализация EN/RU/SR (все три .arb + `flutter gen-l10n`).

## Тесты

- Юнит: группировка по линии+направлению, дедупликация остановок, сортировка.
- Бэкенд: vitest на агрегацию эндпоинта (мок-провайдер).
- Widget-тест экрана с `kMapRenderingEnabled=false` при необходимости.
- `flutter analyze lib test` и `npx tsc --noEmit` чистые.

## Definition of Done

1. На staging экран «Рядом» показывает живой список линий вокруг реальной
   геопозиции; обновление ≤ раз в 30 с.
2. Фиче-флаг `nearby_list` полностью скрывает фичу.
3. Тесты проходят, analyze/tsc чистые.
4. Отчёт `docs/reports/2026-MM-DD-nearby-list.md` (RU): что сделано, выбор по
   [STOP & ASK]-развилкам, что сознательно отложено (learning-sort, пины),
   впечатление — ложится ли формат на наши данные.
5. Отчёт заканчивается разделом **«Как проверить»** для владельца без
   терминала: сессия сама деплоит staging; владельцу — URL, где взять доступ
   к превью (docs/staging.md) и что тыкать / что должно быть видно.
6. Merge в main — только по явной команде владельца.
