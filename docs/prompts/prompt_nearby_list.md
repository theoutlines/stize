# Задача: довести «Рядом» (nearby-list), интегрировав с расписанием-фолбэком

Ветка: `feature/nearby-list` (существующая worktree, отстала на 54 коммита) ·
Отчёт: `docs/reports/2026-07-14-nearby-list.md` (RU)

## Контекст и решение

Экран «Рядом» (GPS → все линии рядом без ввода, nearby-first как у Transit) —
живая неслитая работа: draggable sheet, эндпоинт `/api/v1/arrivals/nearby`, флаги
`nearby_list` + `nearby_sort_board`. Ветка писалась ДО сегодняшнего массива и
сильно отстала.

**Решение владельца — вариант B: не механический merge, а переосмысление.** Раз в
main теперь есть расписание-фолбэк (список Фаза 1 + карта Фаза 2, fan-out по
остановкам) — «Рядом» должна ОПИРАТЬСЯ на эту инфраструктуру (живое + расписание в
списке рядом), а не на свою старую реализацию. Иначе «Рядом» покажет пустые
остановки там, где нет живого — ровно та пустота, которую весь день чинили. «Рядом»
обязана унаследовать расписание, чтобы остановка рядом никогда не была пустой.

## Порядок ([STOP & ASK] на развилках)

### 1. Ребейз/merge main — осознанно, не механически
Ветка конфликтует с main в ~11 файлах, включая **backend `arrivals.ts` /
`transitProvider.ts`**, где main плотно переписан под расписание (schedule fallback,
503-фикс, кэш no-store, fan-out). Разрешать эти конфликты МЕХАНИЧЕСКИ нельзя — можно
внести регрессию в свежестабилизированный прод.
- Merge main в ветку, конфликты backend разбирать по одному, сохраняя сегодняшнюю
  работу (расписание, кэш, 503-фикс) как основу.
- **[STOP & ASK]** на каждом нетривиальном backend-конфликте: показать, что в main,
  что в ветке, как совместить — до разрешения.

### 2. Переосмыслить эндпоинт «Рядом» под новую инфраструктуру
Старая ветка делала свой `/api/v1/arrivals/nearby`. Сейчас в main есть:
- schedule fallback (`getArrivals({includeSchedule})`) — живое + плановые в списке;
- fan-out по остановкам с шаренным 30s-кэшем;
- дедуп живое-первым + scheduled-следующими.
Реши ([STOP & ASK]): «Рядом»-эндпоинт должен ПЕРЕИСПОЛЬЗОВАТЬ `getArrivals` с
расписанием (чтобы остановки рядом несли живое+расписание), а не свою старую логику.
Проверь, не дублирует ли ветка fan-out, который уже есть в main — если да,
переключить на общий. Не плодить две параллельные системы «остановки рядом».
Соблюдать: 30s-кэш не обходить, upstream не долбить, лимиты воркера (помни утренний
503 от протёкшего расписания в map-путь — не повторить в nearby).

### 3. Экран «Рядом» (клиент)
- Draggable sheet «Рядом»: список линия→направление с ETA, живое + scheduled-строки
  (как в списке прибытий остановки — переиспользовать бейджи/дедуп, не изобретать).
- Сортировка «по времени до посадки» за `nearby_sort_board`.
- Привязка к геопозиции (переиспользовать сегодняшний position-стрим, не свой).
- Пустое состояние / нет геолокации — как в остальном приложении.

### 4. Флаги
`nearby_list` + `nearby_sort_board` — OFF прод / ON staging (как всё).

### 5. Галерея
Новый экран «Рядом» + состояния → обновить `design/gallery/` (по DoD) и
передеплоить `preview-gallery`. Правила — `design/gallery/README.md`.

## Границы
- Живой источник / 30s-кэш / схему D1 не трогать.
- НЕ сломать сегодняшнюю работу при разрешении конфликтов: расписание (список+карта),
  кэш no-store, 503-фикс, direction, символьный слой. Они — основа, ветка садится
  поверх.
- Код EN, отчёт RU, без секретов.
- Merge в прод — по явной команде владельца.
- Staging-preview билд — с правильными дефайнами (`API_BASE_URL=staging` +
  `ENVIRONMENT=staging`, урок дня в CLAUDE.md), иначе бьёт в прод.

## DoD
1. main влит в ветку, backend-конфликты разобраны осознанно (не механически),
   сегодняшняя работа цела, тесты зелёные (client + backend).
2. «Рядом»-эндпоинт переиспользует расписание-инфраструктуру (живое + scheduled),
   остановки рядом не пустые; fan-out не дублируется, лимиты воркера соблюдены.
3. Экран «Рядом» работает за флагом, переиспользует position-стрим и бейджи прибытий.
4. Галерея обновлена, тесты + analyze чисты.
5. Отчёт RU + «Как проверить» без терминала (staging-URL + бейдж STAGING).
6. Merge в прод — по команде владельца.
