# Задача: свежесть GTFS — обновление фида + гашение устаревших линий (2 фазы)

Ветка: `feature/gtfs-freshness` · Worktree: `../stigla-gtfs-freshness`
Отчёт: `docs/reports/2026-07-XX-gtfs-freshness.md` (RU)

Две фазы в ОДНОЙ ветке, строго последовательно: сначала обновляем фид (корень),
затем — на цифрах ПОСЛЕ обновления — настраиваем гашение остатка «призраков».
Порядок важен: гашение калибруется под остаток после свежего фида, не под старый.

## Контекст (по итогам аудита линий 2026-07-13)

Живая сверка (`docs/reports/2026-07-13-line-data-audit.md`) показала: на **~21%
активных остановок** в живом потоке есть валидные текущие маршруты (трамвай 8,
E6, KC1, пригород 491/860x/975…), которых **нет в нашем GTFS** (скачан вручную
2025-10-31, 8+ месяцев). Источник и id-пространство корректны — корень в
устаревшем справочнике stop→routes, системно, перекос в пригород/край сети.
Плюс app-side усилитель: экран остановки показывает GTFS ∪ live
([stop_screen.dart:152]), поэтому устаревшие «призраки» висят, даже когда сеть
их сняла.

Код/комментарии EN, отчёт RU, публичный текст без секретов/upstream-деталей
(GTFS с data.gov.rs — открытые данные, его URL в коде допустим; URL/параметры
живого источника — только в env).

# ФАЗА 1 — жизненный цикл фида (корневой фикс)

Ориентир — GTFS Schedule Best Practices (gtfs.org) и гайдлины Transit
(`docs/BENCHMARKS.md`, раздел «Стандарты»): фид живёт итерациями, проверяется
валидатором, потребитель знает и показывает дату данных.

### 1. Скрипт обновления — `backend/scripts/gtfs-refresh.mjs` (`npm run gtfs:refresh`)
1. Скачать актуальный официальный GTFS Белграда с data.gov.rs в gitignored
   `backend/gtfs_raw/` (бэкап предыдущей версии рядом).
   - **[STOP & ASK]** если нет стабильного прямого URL на zip (скрейпинг каталога/
     меняющийся URL) — согласовать способ обнаружения свежей версии.
2. Прогнать канонический **MobilityData GTFS Validator**, сохранить сводку
   (errors/warnings) в вывод.
   - **[STOP & ASK]** если Java-валидатор неподъёмен — предложить облегчённые
     структурные проверки своим кодом и согласовать. Ошибки валидатора НЕ блокируют
     сборку автоматически (фид городской), но сводка обязана попасть в отчёт.
3. **Diff-отчёт** нового фида против текущего бандла `public/gtfs/`: добавленные/
   исчезнувшие `route_id`/`stop_id`, изменение числа направлений/shape, диапазон
   календарей. В консоль + `backend/gtfs_raw/refresh-report.md` (gitignored).
4. По подтверждению — пересборка бандла существующим `npm run gtfs:build`.

### 2. Метаданные свежести — `feed_meta.json`
- При сборке (`build-gtfs.mjs`) генерировать `public/gtfs/feed_meta.json`: дата
  версии фида (из `feed_info.txt` или даты файла), дата сборки, диапазон
  календарей, счётчики (lines/stops/shapes).
- Раздача: **явный Hono-роут** `GET /api/v1/gtfs-meta` (gotcha из CLAUDE.md —
  статик-биндинг обходит CORS).
- Staleness-guard: конец календаря в прошлом или фиду > 6 мес → громкий warning
  в `gtfs:build`.

### 3. Дата данных в продукте
- В About: строка «Route data: <месяц год>» (EN/RU/SR, `flutter gen-l10n`) из
  `/api/v1/gtfs-meta`, тихий фолбэк (нет ответа — строка не показывается).

### 4. Первый реальный refresh + верификация
- Выполнить первый refresh: скачать свежий фид, прогнать валидатор, приложить
  diff-сводку к отчёту. Пересборку — только если diff без ломающих сюрпризов;
  иначе **[STOP & ASK]** со сводкой.
- **Верификация против живого потока:** после обновления повторить массовую
  сверку из аудита (GTFS stop→routes vs живой upstream по выборке остановок),
  показать цифрой, что ~21% пробел закрылся/сократился. Скрипт сверки взять из
  ветки `fix/line-data-audit`, если сохранился, иначе восстановить.
- **Зафиксировать ОСТАТОК призраков после свежего фида** — это вход для Фазы 2.

### 5. Каденция
- Обновление — раз в квартал, в связке с ревизией ростеров (B7) и конкурентов.
  Зафиксировать в `docs/WORKFLOW.md` (раздел периодических рутин — создать, если
  нет) и `docs/BACKLOG.md`.

# ФАЗА 2 — гашение остатка «призраков» за флагом

Делать ПОСЛЕ Фазы 1, откалибровав под реальный остаток. Если после свежего фида
призраков почти не осталось — доложить это и, возможно, Фаза 2 не нужна срочно
(**[STOP & ASK]**).

### Главная тонкость (не рубить сплеча)
**«Снята из сети навсегда» ≠ «сейчас не едет».** Линия может отсутствовать в
живом потоке по невинным причинам: ночь, пересменка, большой интервал, простой.
Гасить её как призрак — ошибка (пассажир перестанет видеть реальную линию).
Механизм КОНСЕРВАТИВНЫЙ: **лучше показать лишнюю линию, чем спрятать реальную.**

### [STOP & ASK] — критерий призрака (до реализации Фазы 2)
Определить и согласовать:
1. Есть ли на уровне остановки `first/last seen` по линии×остановке в
   аналитических агрегатах (копятся с 2026-07-10), годятся ли как сигнал «давно
   не появлялась»?
2. Порог устойчивости (напр.: нет в потоке N дней подряд И по GTFS должны были
   быть рейсы) — предложить, ждать решения.
3. Скрывать «призрака» полностью, или мягко помечать (приглушённо / «нет
   данных»)? Мягкий вариант безопаснее для доверия.

### Реализация (после согласования)
- Классификатор линии остановки: `live` / `scheduled-present` (в GTFS, ожидаемо
  не в потоке сейчас) / `stale-candidate` (в GTFS, давно не в потоке). Чистая
  функция, тестируемая, отдельно от UI.
- Применение на экране остановки ([stop_screen.dart], `stop_sheet.dart`) и в
  чипах — гасить/помечать по выбранному варианту.
- **За фиче-флагом** `stale_line_suppression`: OFF на проде, ON на staging. При
  OFF — текущее поведение (GTFS ∪ live, ничего не гасится).

## Границы
- Политику 30s-кэша, upstream живых данных, SWR, схему D1/KV — не трогать.
- НЕ смешивать с `feature/symbol-layer` (рендер) — та в других файлах; если
  Фаза 2 трогает `stop_screen.dart`, а symbol-layer тоже — это ЕДИНСТВЕННОЕ
  пересечение; при мерже сверить порядок с владельцем.
- Консервативность гашения превыше полноты: ложно скрыть реальную линию хуже,
  чем оставить призрака.
- `main` релизопригоден; всё за флагами / аддитивно.

## Definition of Done (CLAUDE.md)
1. Код в ветке; тесты зелёные. Тесты: генерация `feed_meta.json`, diff-логика
   (чистые функции), классификатор линий (включая кейс «ночью не едет ≠ призрак»).
2. Отчёт RU: Фаза 1 (diff первого refresh, вывод валидатора, результат сверки —
   было ~21%, стало X%), Фаза 2 (критерий призрака, что гасится/помечается) или
   обоснование, почему Фаза 2 отложена.
3. BACKLOG/WORKFLOW: квартальная каденция; статусы; связь с аудитом.
4. Меняется About (Фаза 1) и, возможно, экран остановки (Фаза 2) → обновить
   `design/gallery/` (about, stop-arrivals) + передеплой preview-gallery.
5. «Как проверить» без терминала: сессия деплоит staging-превью сама; владельцу —
   URL, `/api/v1/gtfs-meta` в браузере, About со строкой даты, и какие остановки
   сверить с эталонами (где раньше висели призраки).
6. Merge в main — по явной команде владельца.
