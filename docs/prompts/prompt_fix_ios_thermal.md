# Задача: перегрев iPhone при использовании веб-версии — диагностика и фикс

Ветка: `fix/ios-web-thermal`. Отчёт: `docs/reports/YYYY-MM-DD-ios-thermal.md` (RU).
Код и комментарии — EN. Текст публичный: без секретов и инфра-деталей.

## Симптом

На iPhone 14 Pro (Safari, stigla.theoutlines.xyz) телефон ощутимо нагревается
в течение нескольких минут обычного использования: открытая карта с живыми
машинами. Это признак постоянной высокой загрузки CPU/GPU — скорее всего,
приложение рисует непрерывно на 60fps там, где могло бы простаивать.

## Задача

Двухэтапная: сначала **измерить и локализовать**, потом чинить только
подтверждённое. Не начинать с оптимизаций «наугад».

### Этап 1 — диагностика

1. Собрать профиль web-сборки: Chrome DevTools Performance (desktop как
   прокси) + `flutter run -d chrome --profile` с DevTools timeline.
   Зафиксировать: FPS в покое (карта открыта, никто не трогает), % времени
   в raster/UI thread, что триггерит кадры.
2. Проверить гипотезы в порядке подозрительности:
   - **H1. Непрерывный ticker анимации машин.** `vehicle_track_animator` /
     подписанные на него виджеты: продолжает ли тикать, когда (а) все машины
     доехали до последнего реального фикса и стоят, (б) вкладка скрыта /
     приложение свёрнуто, (в) маркеры вне вьюпорта?
   - **H2. WidgetLayer-маркеры.** Каждый `VehicleMarker` — живой Flutter-виджет,
     привязанный к геоточке. Проверить, не перерисовывается ли всё дерево
     маркеров каждый кадр (в т.ч. при неподвижной карте), нет ли лишних
     rebuild'ов из-за провайдеров/setState на каждый тик анимации.
   - **H3. Нет паузы при скрытой странице.** `visibilitychange` / lifecycle:
     останавливаются ли тикеры, поллинг и подписки, когда вкладка в фоне?
   - **H4. Разрешение канваса.** iPhone 14 Pro — DPR 3. Проверить фактический
     размер канваса и стоимость растеризации; оценить эффект ограничения DPR
     до 2 для карты/canvas (только оценить, решение — через STOP & ASK).
   - **H5. Рендерер.** Какой web-рендерер используется в проде (canvaskit /
     skwasm / html) и есть ли известные проблемы связки с MapLibre на iOS Safari.
3. Записать в отчёт таблицу: гипотеза → подтверждена/нет → доля вклада (оценка).

### Этап 2 — фиксы (только по подтверждённым гипотезам)

Ожидаемые направления (не предписание — по результатам этапа 1):

- Ticker останавливается, когда все интерполяции дошли до целевых точек;
  просыпается по новому фиксу данных. «Idle = ноль кадров».
- Полная пауза анимаций и поллинга при `hidden` (visibility API /
  `AppLifecycleState`), возобновление при возврате.
- Маркеры вне вьюпорта не анимируются (позиция обновляется без анимации
  при входе в вьюпорт).
- Изоляция перерисовок: `RepaintBoundary` вокруг маркеров, rebuild только
  того маркера, чья позиция изменилась, а не всего слоя.

Ограничения:
- Клиентский поллинг остаётся ≥30s (см. CLAUDE.md) — его не трогаем.
- Плавность движения машин на переднем плане не должна заметно деградировать.
- Эвристика «застрял» не должна ложно срабатывать из-за пауз (учесть
  замороженное время при возобновлении).

## [STOP & ASK]

- Если главный вклад — H4/H5 и фикс требует смены рендерера или ограничения
  DPR: остановиться, показать замеры и трейдоффы (чёткость vs нагрев),
  ждать решения владельца.
- Если для существенного выигрыша нужно пожертвовать плавностью интерполяции
  (например, снизить FPS анимации машин до 30) — предложить варианты, ждать.
- Если проблема воспроизводится только на реальном iOS Safari и десктопный
  профиль её не ловит — предложить план замеров на устройстве (Safari Web
  Inspector по кабелю), ждать подтверждения.

## Definition of Done

1. В отчёте: замеры «до» (FPS/CPU в покое и при движении карты) и «после»,
   по одинаковой методике; таблица гипотез.
2. В покое (карта открыта, машины стоят, пользователь не трогает экран)
   приложение не рисует кадры непрерывно.
3. При скрытой вкладке — нулевая активность (нет тикеров, нет поллинга).
4. `flutter analyze` чистый, тесты проходят; новые юнит-тесты на логику
   остановки/возобновления тикера.
5. Merge в main — только по явной команде владельца.

## Верификация владельцем

После деплоя на preview: 10 минут обычного использования на iPhone 14 Pro —
субъективный нагрев + проверка через Safari Web Inspector (Timelines → CPU),
что в покое загрузка близка к нулю.
