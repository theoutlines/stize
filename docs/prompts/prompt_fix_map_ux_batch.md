# Задача: батч фиксов карты и UI (fix/map-ux-batch)

## Контекст

Владелец прогнал прод (stigla.theoutlines.xyz) на iOS Safari и десктопе и собрал
пачку багов/UX-проблем. Все — про карту, маркеры ТС и мелкий UI. Референс
поведения в нескольких местах — Яндекс.Карты (описано словами, скриншоты в репо
не кладём).

Ветка: `fix/map-ux-batch`. Код/комментарии — EN, отчёт — RU в
`docs/reports/YYYY-MM-DD-fix-map-ux-batch.md`. Перед началом прочитать
`CLAUDE.md` (особенно gotchas про polling ≥30s, `AsyncValue.valueOrNull`,
`kMapRenderingEnabled` в тестах).

Фиксы независимые — коммитить по одному (F1..F9 в сообщении коммита),
чтобы можно было откатывать точечно.

---

## F1. Маркер ТС «плывёт» мимо дорог

**Симптом:** точка ТС движется по прямой через парки/кварталы, игнорируя
геометрию маршрута. Наблюдается периодически в разных местах. Это уже
чинили — видимо, не до конца.

**Ожидание:** интерполяция маркера идёт вдоль полилинии маршрута
(`all_stations`), а не по прямой между фиксами.

**Куда смотреть:** `core/vehicle_track_animator.dart` (интерполяция),
`core/vehicle_route.dart` (проекция на маршрут). Проверить edge cases:
фикс, спроецированный не на тот сегмент (петли маршрута, близкие параллельные
участки), ТС с пустым/битым `all_stations`. Написать unit-тесты на найденный
сценарий регрессии — аниматор специально вынесен в чистую математику.

## F2. Тап по ТС: не отдалять камеру до всего маршрута

**Симптом:** при выборе автобуса/троллейбуса карта делает fitBounds на весь
маршрут — пользователя выкидывает с его зума.

**Ожидание (как в Яндексе):** камера остаётся где была (максимум — мягко
доцентрировать ТС, если он у края). Маршрут подсвечивается целиком как слой;
если пользователь сам отдалит — увидит его полностью. Никакого автоматического
изменения зума.

## F3. iOS Safari: ложное «Location is off» + viewport не на весь экран

Два подпункта, оба видны на мобильном Safari:

**3a. Геолокация.** Разрешение на локацию в Safari выдано, но приложение
показывает баннер «Location is off. Allow location access…».
Разобраться, какой именно error code приходит от Geolocation API
(PERMISSION_DENIED vs TIMEOUT vs POSITION_UNAVAILABLE) и не мапим ли мы все
ошибки на один текст «выключено». Сообщения развести по причинам; при
timeout/unavailable — не врать про выключенную локацию.

**3b. Viewport.** Сайт занимает не весь экран: сверху полоса под статус-баром
(часы/заряд), контент обрезан этой областью. Нужно `viewport-fit=cover` +
`env(safe-area-inset-*)` паддинги: карта — под статус-бар на весь viewport,
элементы управления (бургер, геолокация, поиск, чипы) — с отступами, чтобы
не уплывали под системные зоны. Проверить и notch, и home indicator снизу.

## F4. Несколько ТС в одной точке + стрелки слиты с пиллом

**Симптом:** у маркера «56N» две стрелки в разные стороны — вероятно, в одной
точке стоят два ТС, отрисованные друг на друге. Плюс стрелки направления
визуально сливаются с баблом ТС в один кривой блоб.

**Ожидание:**
- ТС с (почти) совпадающими координатами раздвигать (spiderfy/offset по кругу
  на несколько пикселей), чтобы читалось «здесь их несколько».
- Стрелку направления визуально отделить от пилла (позиция/отступ/слой) —
  сейчас она рисуется вплотную под ним.

**Куда смотреть:** `core/map_support.dart` (`VehicleMarker`),
`home_map_screen.dart` / `widgets/live_vehicles_map.dart` (рендер маркеров).

## F5. На дальнем зуме транспорт исчезает совсем

**Симптом:** при сильном отдалении (город целиком) ТС перестают показываться —
карта пустая. Референс: Яндекс на том же зуме показывает десятки точек по
всему городу (мелкими, без пиллов).

**Ожидание:** транспорт виден на любом зуме. На дальних зумах — деградация
представления (точки вместо пиллов, прореживание), но не исчезновение.
Медленное движение точек по городу — само по себе ценная картинка.

**Важно, сначала диагностика:** выяснить, где именно обрезается — клиентский
zoom-threshold рендера или то, что `GET /vehicles/nearby` физически не может
покрыть весь город (fan-out ограничен ≤12 остановок / ≤1500 м, upstream —
только per-stop). Если ограничение клиентское — снять его и сделать
деградацию представления. Если данных на весь город просто нет:

**[STOP & ASK]** — предложить варианты (например: показывать на дальнем зуме
только то, что уже есть в кеше вокруг последних viewport'ов; расширенный
fan-out с жёстким бюджетом запросов; честный UI-хинт «приблизьте карту, чтобы
увидеть транспорт») с оценкой нагрузки на upstream. Ничего не менять в
политике 30s-кеша без явного решения владельца.

## F6. Карточка остановки: пустой чип линии и фантомное прибытие

**Симптом (остановка Tašmajdan):** в фильтре линий три чипа — «All lines»,
«24» и пустой третий; в списке прибытий — строка с иконкой автобуса и «Now»,
но без номера линии и направления.

**Ожидание:**
- Чипы рендерятся только для непустых линий; пустые значения из данных
  отфильтровываются (и залогировать/понять, откуда приходит пустая линия —
  вероятно, junk из upstream).
- Zero-case: если прибытий нет — явное состояние «Пока нет прибытий»
  (локализовано EN/RU/SR), а не пустая строка с иконкой.
- Фантомные записи без номера линии не показывать.

## F7. Чипы сохранённых остановок обрезаются по краям

**Симптом:** горизонтальный ряд чипов сохранённых остановок режется у левого
и правого краёв экрана — чипы обрезаны, а не уходят в скролл до края.

**Ожидание:** горизонтальный скролл с корректными edge-паддингами: первый и
последний чип докручиваются полностью, ничего не клипается. Учесть
safe-area (связано с F3b).

## F8. Поиск линии: показывается только одно направление

**Симптом:** поиск «79» выдаёт только «Dorćol /SRC Milan Gale Muškatirović/ →
Mirijevo 4», хотя линия ходит в обе стороны — обратного направления в выдаче
нет.

**Ожидание:** обе стороны линии в выдаче (или одна запись с выбором
направления — по текущему паттерну UI, не изобретать новый).

**Дополнительно:** пройтись по поиску в целом — индексация направлений,
дубликаты, регистр/латиница-кириллица. Написать тесты на «линия ищется в оба
направления» хотя бы для пары линий.

## F9. Убрать настройку интервала обновления

**Симптом:** в настройках выбор интервала обновления: 15 / 30 / 60 секунд.
Бэкенд обновляет upstream-снапшот раз в ~30 с (SWR-кеш, глобально на ключ),
поэтому 15 с — вредно (клиент дважды читает идентичные позиции, heuristic
движения считает ТС «застрявшим»), а 60 с не даёт заметной выгоды.

**Ожидание:** настройку удалить целиком. Интервал поллинга — константа 30 с
(держать согласованной с бэкенд-кешем, см. gotcha в CLAUDE.md). Сохранённое
пользовательское значение игнорировать/вычистить при миграции, UI настройки
и связанный код убрать. Если от настройки зависят тесты — обновить.

1. Все фиксы, не упершиеся в [STOP & ASK], закоммичены отдельными коммитами
   в `fix/map-ux-batch`; `flutter analyze` и тесты зелёные.
2. Новые unit-тесты: F1 (интерполяция вдоль маршрута), F8 (оба направления
   в поиске). Для F6 — тест фильтрации пустых линий, если логика в Dart.
3. Отчёт RU в `docs/reports/` с разбивкой по F1–F9: что нашли, что поменяли,
   что осталось (включая исход [STOP & ASK] по F5).
4. Merge в main — только по явной команде владельца.
