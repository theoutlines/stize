# Задача: слой покрытия на основной карте при зум-ауте

Ветка: `feature/coverage-on-main-map`, worktree `../stigla-coverage-on-main-map`.
Контекст: `CLAUDE.md`, `docs/BACKLOG.md`, `docs/COVERAGE_MAP.md` — вкладка
Coverage уже в main (за флагом `coverage_map_show`), её heatmap-слой и данные
переиспользуются здесь. Прочитать реализацию вкладки до начала работы.

## Идея

На основной карте при сильном зум-ауте отдельные остановки не несут информации
(сейчас они схлопываются в кластеры с числами). Вместо кластеров на дальнем
зуме показываем heatmap покрытия — тот же слой, что на вкладке Coverage:
пользователь видит, «где транспорт вообще есть», а при приближении карта
возвращается к обычному режиму с остановками.

Это пассивный фоновый слой и одновременно дискавери вкладки Coverage.

## Поведение

1. **Зум-порог с гистерезисом.** Ниже порога — heatmap, выше — остановки.
   Порог подобрать визуально (ориентир: зум, на котором сейчас появляются
   кластеры вместо отдельных остановок); гистерезис ~0.5 зума, чтобы слои
   не мигали при колебании вокруг порога.
2. **Плавный переход.** Crossfade по зуму: кластеры/остановки растворяются,
   heatmap проявляется (интерполяция opacity обоих слоёв в переходной зоне,
   не жёсткое переключение).
3. **Живые ТС остаются поверх** heatmap в текущем виде (редкие точки +
   подсказка «приблизь» — поведение не менять).
4. **Слой пассивный.** Никаких контролов, легенды и фильтров на основной
   карте. Тап по heatmap ничего не делает в V0. Интерактив живёт во вкладке
   Coverage.
5. **Lazy load.** GeoJSON точек покрытия грузится только при первом
   пересечении зум-порога (или переиспользуется, если уже загружен вкладкой
   Coverage — общий кеш/провайдер, не два независимых fetch).
6. **Темы.** Слой уже theme-synced на вкладке — здесь то же поведение.

## Реализация

- Переиспользовать источник и стилевые параметры heatmap из вкладки Coverage —
  вынести общее (источник, рампа, radius/intensity expressions) в один модуль,
  не копировать. Расхождение параметров двух мест — баг по определению.
- Параметры heatmap на основной карте могут быть чуть скромнее (ниже
  intensity/opacity), чтобы слой читался как фон, а не как главный контент —
  подобрать визуально.
- [STOP & ASK] если crossfade кластеров технически конфликтует с текущей
  логикой кластеризации остановок (например, кластеры рисуются не слоем,
  а виджетами) — показать варианты с trade-offs.

## Фиче-флаг

`coverage_on_main_map` (KV, env-aware): OFF на проде, ON на staging.
Независим от `coverage_map_show` (вкладка может быть выключена, слой — включён,
и наоборот).

## Тесты и приёмка

- Юнит: логика порога/гистерезиса (чистая функция зум → видимость слоёв).
- Widget-тест основной карты с `kMapRenderingEnabled=false` не ломается.
- `flutter analyze lib test` чист; существующие тесты зелёные.
- Производительность: переход по зуму не вызывает джанк (профилировать
  на web-сборке); heatmap не перерисовывает основную карту лишний раз.

## Definition of Done

1. На staging: зум-аут на основной карте → кластеры плавно сменяются
   heatmap-свечением; зум-ин — обратно; живые ТС видны всегда.
2. Код закоммичен в `feature/coverage-on-main-map`, тесты проходят.
3. Отчёт RU в `docs/reports/2026-07-XX-coverage-on-main-map.md`,
   в конце — раздел «Как проверить» без терминала: staging задеплоен
   сессией, владельцу — URL, доступы, что тыкать.
4. `docs/BACKLOG.md`: пункт «Покрытие на основной карте» → 🚧 за флагом.
5. Merge в main — только по явной команде владельца.

## Ограничения

- Публичный репозиторий: без секретов и инфра-деталей.
- Бэкенд не трогаем — данные уже раздаются.
- Не выходить за рамки: тап по heatmap, переход во вкладку Coverage по тапу,
  фильтры на основной карте — вне задачи (кандидаты на следующие итерации).
