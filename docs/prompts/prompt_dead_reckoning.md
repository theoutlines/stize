# Задача: dead reckoning — непрерывное движение маркеров ТС между обновлениями

Ветка: `feature/dead-reckoning` · worktree: `../stigla-dead-reckoning`
Отчёт: `docs/reports/YYYY-MM-DD-dead-reckoning.md` (RU). Код и комментарии — EN.

## Контекст

Данные о позициях приходят раз в ~30 с (клиентский поллинг выровнен на backend-кеш,
см. CLAUDE.md — не ускорять). Текущий аниматор (`core/vehicle_track_animator.dart`)
консервативен: маркер догоняет последний реальный фикс и никогда не едет дальше.
Итог — маркер стоит 30 секунд, потом прыгает.

Эталонное поведение (так делает Яндекс при той же 30-секундной частоте данных):
**экстраполяция**. После фикса метка продолжает ехать вперёд по геометрии маршрута
с оценённой скоростью; новый фикс не телепортирует её, а мягко корректирует.
Цена — метка может временно врать на десятки метров; это осознанный трейд-офф.

Вся математика остаётся в `vehicle_track_animator.dart` (чистая, unit-testable),
геометрия маршрута — из `all_stations` через `core/vehicle_route.dart`.

## Что сделать

### 1. Оценка скорости
- Скорость вдоль маршрута из двух последних фиксов: расстояние по полилинии
  маршрута (не по прямой) / разница времени между фиксами.
- Клампить в разумные границы (константа, ориентир 0–60 км/ч).
- Одного фикса недостаточно → скорость неизвестна → поведение как сейчас
  (стоим на фиксе). Никакой экстраполяции с потолка.

### 2. Экстраполяция
- Достигнув последнего фикса, маркер продолжает двигаться вперёд вдоль
  полилинии маршрута с оценённой скоростью.
- Кап: не дальше N секунд от timestamp фикса (константа, ориентир 45 с).
  По достижении капа — плавное торможение до остановки, дальше работает
  существующая эвристика «застрял».
- Экстраполяция не пересекает конечную остановку маршрута.

### 3. Коррекция при новом фиксе — без телепорта
- Маркер сходится к актуальной траектории за несколько секунд изменением
  скорости, а не прыжком.
- **Маркер никогда не движется назад.** Если экстраполяция обогнала реальный
  фикс — тормозим/стоим и ждём, пока реальность догонит.
- Фикс «позади» предыдущего (GPS-шум) — игнорировать для оценки скорости.

### 4. Переключатель
- Флаг в духе `kMarkerDeadReckoning` рядом с `kMapRenderingEnabled`
  (compile-time константы достаточно), позволяющий вернуть старое поведение.
  Нужен для сравнения на staging и как страховка.

### 5. Тесты
- Юнит-тесты на: оценку скорости (в т.ч. кламп и фикс назад), кап экстраполяции,
  сходимость после нового фикса без движения назад, деградацию при одном фиксе
  и при отсутствии геометрии маршрута.

## Границы задачи
- Не трогать частоту поллинга и backend.
- Не моделировать остановки на маршруте (замедление у остановок) — вне scope,
  упомянуть в отчёте как возможное улучшение.
- UI маркера (пилюля, классификация) не меняется — только движение.
