# Отчёт: карта покрытия транспортом (Coverage Map, V0)

Дата: 2026-07-12 · Ветка: `feature/coverage-map` · Спека: `docs/COVERAGE_MAP.md`

> **Важно:** разделы ниже описывают исходный подход (line layer со схлопыванием
> сегментов). По ходу задачи модель рендера эволюционировала до **тепловой карты
> (MapLibre heatmap)**. Актуальное итоговое состояние — в разделе
> [«Финальное состояние»](#финальное-состояние-shipped) в конце файла; спека
> `docs/COVERAGE_MAP.md` обновлена под факт.

## Что сделано

Новая вкладка-инфографика «Покрытие»: светящиеся коридоры маршрутов поверх
theme-synced подложки. Слой статичный, предвычисленный, за фиче-флагом.

### 1. Пайплайн данных (`backend/`)
- `scripts/coverage-core.mjs` — чистое ядро (без I/O): схлопывание сегментов и
  счёт маршрутов. Логика:
  1. каждая точка shape квантуется в ячейку сетки (`DEFAULT_GRID_DEG = 0.0005°`,
     ≈ 55 м с-ю / 40 м з-в на широте Белграда);
  2. соседние точки → **ненаправленный** сегмент между ячейками (A→B и B→A
     совпадают — оба направления линии схлопываются в один коридор);
  3. на сегмент копится набор **различных линий** (`routes_count`) и типов ТС;
  4. сегменты с одинаковым набором линий сшиваются в полилинии (жадный обход
     рёбер) → резкое сокращение числа фич; near-collinear точки прорежаются
     (RDP, ε = 1 ячейка).
- `scripts/build-coverage.mjs` — runner: читает `public/gtfs/lines.json` +
  `shapes/*.json`, пишет `public/gtfs/coverage.geojson`. Подключён к
  `npm run gtfs:build` (и отдельно `npm run coverage:build`).
- Properties фич: `routes_count` (int, вес V0) и `types` (array:
  tram/trolleybus/bus, стабильный порядок). Клиент читает вес из **одного
  именованного property** (`_weightProperty`), поэтому будущие веса (frequency,
  observed_intensity — V1/V2) добавляются пересборкой файла без правок клиента.

**Выбор допуска сетки.** Метод схлопывания (grid-snap + ненаправленный ключ)
чистый (без двоящихся линий/разрывов), поэтому [STOP & ASK] не потребовался.
Единственный knob — размер ячейки; замерил зависимость и выбрал `0.0005°` как
баланс «чисто + легко»:

| grid | фич | raw | gzip |
|------|-----|-----|------|
| 0.00025° | 48 605 | 7.4 МБ | 472 КБ |
| 0.0004° | 26 570 | 3.9 МБ | 240 КБ |
| **0.0005°** | **17 981** | **2.66 МБ** | **160 КБ** |
| 0.0006° | 12 790 | 1.9 МБ | 125 КБ |

Компромисс: чем крупнее ячейка, тем лучше сливаются коридоры и легче файл, но
заметнее «лесенка» на глубоком зуме. Для инфографики масштаба города при камере
«весь Белград» 55 м — субпиксельно; ноль двоения важнее гладкости.

### Итоговый файл
`public/gtfs/coverage.geojson`: **17 981 фич, 2.66 МБ raw / ~160 КБ gzip**.
Вес `routes_count` от 1 до 33. Порог спеки (≤ ~1–2 МБ gzip) с запасом соблюдён.

### 2. Раздача (`backend/`)
- `GET /api/v1/coverage` — явный Hono-роут (не сырой `/gtfs/*`), чтобы получить
  CORS-заголовки для web-сборки с другого origin (gotcha из CLAUDE.md).
  Стримит статик-ассет `/gtfs/coverage.geojson`, `cache-control: max-age=86400`.
- Проверено локально (`wrangler dev`): `200`, `Access-Control-Allow-Origin: *`,
  `content-type: application/json`, тело — валидный FeatureCollection.

### 3. Экран (`app/`)
- `presentation/screens/coverage_screen.dart` — третий экран `IndexedStack`
  (карта / идеи / покрытие), пункт в drawer после «Идей», роут `/coverage`.
- Подложка — `core/map_style.dart` (theme-synced), камера — весь Белград,
  независима от основной карты. Смена темы → `setStyle` → слой пересоздаётся.
- Слой: `GeoJsonSource(data: '$apiBaseUrl/api/v1/coverage')` + `LineStyleLayer`
  с expression’ами:
  - `line-width`: interpolate по `routes_count` (0.6 → 8 px);
  - `line-color`: тёплая шкала (ember → белый) на тёмной теме, сине-навигаторная
    на светлой; при выборе **ровно одного** типа — брендовый цвет типа из
    `map_support.dart` (фильтр читается по оттенку), вес остаётся в ширине/альфе;
  - `line-opacity`: слабые коридоры чуть тусклее;
  - `line-blur: 0.4` — лёгкое свечение (свойство line-слоя, не растровый blur).
- Чипы фильтра (мультивыбор: Все / Трамвай / Троллейбус / Автобус) меняют слой
  через remove+add layer (в 0.3.x StyleController нет setFilter/setPaint);
  **источник не пересобирается**, повторного fetch нет.
- Легенда веса: клиновидная шкала dim→bright с подписями «меньше … больше».
- `kMapRenderingEnabled=false` → плейсхолдер (для тестов).

### 4. Фиче-флаг
`coverage_map_show` добавлен в `FEATURE_FLAGS` (KV, env-aware): OFF на проде,
ON на staging (как `analytics_show`). Прячет пункт drawer, роут и саму секцию
`IndexedStack` (при OFF экран не строится — карта не создаётся впустую).

### 5. Локализация
Ключи `navCoverage`, `coverageFilterAll`, `coverageLegend*`, `coverageUnavailable`
добавлены в EN/RU/SR + `flutter gen-l10n`. Метки типов ТС переиспользуют
существующие `vehicleType*`.

## Как проверить визуально (iPhone / web)
Ключ MapTiler в репозитории отсутствует (`app/dart_defines.json` gitignored) —
рендер карты локально не снимал. Для приёмки:

1. Backend со свежим ассетом: `cd backend && npm run gtfs:build && npm run dev`
   (роут `/api/v1/coverage` уже раздаёт файл; на staging флаг ON по умолчанию).
2. App на staging-контур (флаг `coverage_map_show` там ON):
   `cd app && flutter run --dart-define-from-file=dart_defines.json \
     --dart-define=ENVIRONMENT=staging`
   (или локальный воркер: `--dart-define=API_BASE_URL=http://localhost:8787`).
3. Открыть drawer → «Покрытие». Ожидаемо: густые коридоры (центр, бул.
   Краља Александра, мосты) очевидно **ярче и толще** одиночных линий на
   окраинах.
4. Чипы: «Трамвай» → остаётся только трамвайная сеть, красным; «Автобус» —
   синим; «Все» — тёплая/синяя шкала по весу.
5. Тема: переключить светлую/тёмную в настройках — подложка и шкала меняются
   вместе. Скриншот тёмной темы — самый «шеринговый».

## Открытые вопросы / заметки
- **Глубокий зум** показывает «лесенку» (следствие сетки 0.0005°). Для V0
  инфографики ок; сглаживание — кандидат в полировку (Strava-стиль растровый
  glow — отдельная задача по спеке).
- **Цвет при мультивыборе 2 типов** падает на весовую шкалу (не по типу) —
  осознанно: сегмент может нести оба типа, однозначного оттенка нет.
- **Legend диапазоны** заданы порогами `[1,3,6,12,20,33]` под текущее
  распределение; при пересборке фида с сильно другим max стоит перепроверить.
- V1/V2 (частота, час дня) вне V0 — структура GeoJSON к ним готова.

## Финальное состояние (shipped)

Итоговая модель рендера — **тепловая карта (MapLibre `HeatmapStyleLayer`)**, а не
дискретные линии. Путь к ней (для истории): line layer со схлопыванием сегментов
→ сырые полупрозрачные линии → выяснилось, что line-слой **не суммирует альфу**,
контраст плотности не возникает → переход на heatmap по облаку точек.

### Данные
- **Рендер-файл** `public/gtfs/coverage.geojson`: каждый GTFS-shape нарезан в
  точки с шагом **~90 м** (`scripts/build-coverage-points.mjs` →
  `buildCoveragePoints`). Point-фичи с property `type`. Наложение маршрутов =
  плотность точек = вес; heatmap суммирует её в GPU, отдельный подсчёт не нужен.
  Размер: **~80 607 точек, ~8.6 МБ raw / ~0.5 МБ gzip**.
- **Задел на будущее** `public/gtfs/coverage-weighted.geojson`
  (`scripts/build-coverage.mjs`): схлопнутые сегменты с `routes_count` + `types`.
  **Вне рендер-пути**, не раздаётся — под будущие веса (V1/V2).
- Оба собираются из `npm run gtfs:build` (и `npm run coverage:build`).

### Раздача
- `GET /api/v1/coverage` (Hono, CORS) стримит `coverage.geojson`.
  Кэш: `public, max-age=3600, s-maxage=60` (короткий edge-TTL, чтобы редеплой
  данных долетал за минуту). Клиент добавляет `?rev=` к URL источника, чтобы
  обходить залипшую CDN-запись при смене модели данных (сейчас `rev=3`).

### Параметры heatmap (client `coverage_screen.dart`)
Откалиброваны офлайн-симулятором математики heatmap (гауссово ядро/накопление/
рампа) по критерию «три уровня на виде весь-Белград».

- **Рампа по `heatmap-density`** (тёмная тема) — растянута, белый только в
  верхних ~7 %:
  `0.0 → rgba(0,0,0,0)` · `0.08 → rgba(60,24,4,.5)` · `0.4 → #8c370c` ·
  `0.7 → #d65a1a` · `0.85 → #ef7b22` · `0.93 → #ffb860` · `1.0 → #ffffff`.
  Светлая тема — синий аналог (`… → #08306b`).
- **`heatmap-radius`** (px, по зуму): `z11→9, z13→8, z15→7, z18→6`
  (умеренный — коридоры не слипаются в кляксы без реальных пересечений).
- **`heatmap-intensity`** (по зуму): `z11→0.024, z13→0.07, z15→0.28, z18→1.0`
  (растёт ~×2 на уровень зума, компенсируя разрежение точек на пиксель; на
  overview низкая, чтобы белым горело только ядро).
- **`heatmap-opacity`**: `z11→0.9, z16→0.85, z18→0.65` (лёгкий уход, чтобы у
  земли проступала подложка).
- Фильтр по типу — `filter: ["any", ["==",["get","type"], <t>], …]` на том же
  слое (пустой выбор = все).
- Легенда — градиент плотности без чисел («реже → чаще»), повторяет рампу.

### Известные ограничения
- На **очень глубоком зуме** точки (шаг 90 м) перестают перекрываться и heatmap
  становится «бусинами» — неизбежно для фикс-шага; heatmap — инструмент
  среднего/дальнего зума (это и основной режим). Лечится двумя источниками
  (грубый/детальный) при необходимости — задел описан в спеке.
- Скриншоты в CI/агенте недоступны; тюнинг делался офлайн-симулятором —
  перенос параметров в реальный GL корректен, но финальную яркость ядра стоит
  глазами подтвердить на устройстве.

## Как проверить

**Staging (флаг ON по умолчанию):** открыть превью → drawer → «Покрытие»,
вид «весь Белград».
- Ожидаемо (Strava-вайб): тёмная карта, неоновые оранжевые прожилки, **белое
  ядро** в центре (+ мостовой/трамвайный спайн), окраины — тусклый тёмно-оранжевый;
  дыры остаются тёмными. Различимы три уровня плотности.
- Чипы «Трамвай/Троллейбус/Автобус» сужают heatmap до выбранных типов; «Все» —
  всё. Переключение темы приложения меняет подложку и рампу.

**Prod:** фича за флагом `coverage_map_show=OFF` — пункт скрыт. Включение без
пересборки: `POST /api/v1/admin/flags {flag:"coverage_map_show", value:true}`
(с админ-токеном) либо KV-ключ `flag:coverage_map_show=1`.

**Локально:** `cd backend && npm run gtfs:build && npx wrangler dev --env staging`
(флаг ON на staging-env), затем
`cd app && flutter run -d chrome --dart-define-from-file=dart_defines.json
--dart-define=API_BASE_URL=http://localhost:8787 --dart-define=ENVIRONMENT=staging`.

**Пересобрать данные:** `cd backend && npm run coverage:build` (шаг нарезки —
`STEP_METRES` в `scripts/build-coverage-points.mjs`).
