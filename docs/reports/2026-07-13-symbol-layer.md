# Отчёт: символьный GPU-слой для движущихся объектов

Ветка: `feature/symbol-layer`. Задача: `docs/prompts/prompt_symbol_layer.md`.
Дата: 2026-07-13. Статус: **готово, за флагом `symbol_layer` (OFF prod / ON staging), в main не мержим.**

---

## Коротко

Движущиеся ТС на главной карте переведены с per-vehicle Flutter-виджетов
(`WidgetLayer`/`VehicleMarker`) на **один батч-слой MapLibre на GPU** по
GeoJSON-источнику. Стоимость рендера теперь **сублинейна** по числу объектов
(40 или 400 символов ≈ одна цена) вместо линейной у виджетов (замеры донора:
мало ТС → 60 fps, ~40 ТС → 19 fps). Слой спроектирован под **типизированные
движущиеся объекты** — не «слой автобусов»: тип объекта (`MovingObjectKind`) и
весь его рендер выбираются data-driven выражениями MapLibre, так что метро,
электрички и микромобильность добавляются одним цветовым правилом + иконкой, без
переписывания слоя.

Всё за флагом `symbol_layer`. При OFF (прод) работает прежний виджетный путь —
**он не удалён, это фолбэк**. Позиции двигает timed-математика A′ из донора
`feature/timed-trajectory` — переехала целиком, поменялся только приёмник:
координаты пишутся в GeoJSON-источник (`updateGeoJsonSource`), карта
интерполирует символы сама.

## Архитектура слоя

Ядро — `app/lib/core/moving_object_layer.dart` (чистая логика + спеки слоёв,
отдельно от карты, полностью юнит-тестируемо).

### Типизированная модель
- `enum MovingObjectKind { bus, tram, trolleybus, metro, train, scooter, bike }`
  — текущие три типа + зарезервированные под роадмап. `fromVehicleType` мапит
  сегодняшний `VehicleType`; будущие типы придут со своим источником данных.
- `class MovingObject { key, position, kind, label, heading, selected, stuck }`.
  **Fleet-ID и идентичность на маркере нет** — только то, что нужно карте нарисовать
  и тапу «доехать» до линии. `label` — номер линии (общий на будущее: у другого
  типа это может быть номер поезда/рейса). `key` (гаражный номер) — для трекинга,
  спайдерфай и тапа, **не показывается**.

### Источник и слои (один GeoJSON-источник `moving-objects-src`, три слоя)
1. **`CircleStyleLayer` (badge)** — цветная «монета»/точка. `circle-color` —
   `match` по `kind` (цвета из `map_support.dart`; stuck → кримсон), `circle-radius`
   растёт с зумом (дальняя точка → монета), кольцо белое, у выбранного —
   толстое тёмное. Это же единственная цель тапа на всех зумах.
2. **`SymbolStyleLayer` (arrow)** — стрелка направления, `icon-rotate: ['get','heading']`,
   смещена наружу вдоль движения, `icon-rotation-alignment: map`. Только на детальном
   зуме; отфильтрована при `heading == 0` (неизвестно), чтобы не рисовать ложный север.
3. **`SymbolStyleLayer` (label)** — иконка типа над номером линии (`text-field:
   ['get','label']`), оба поверх монеты. Только на детальном зуме.

Прогрессивная детализация (дальний зум = точки, ближний = монеты со стрелкой и
номером) — целиком на GPU через `minzoom` слоёв и zoom-выражение радиуса, **без
Flutter-rebuild на зуме**.

Расширяемость (ключевое требование): **добавить тип = добавить арм в `match` по
`kind` + зарегистрировать иконку.** Источник, слои, тап и спайдерфай не меняются.
Объекты из других источников (не GSP) вливаются в тот же слой, если мапятся на `kind`.

### Что чистое и покрыто тестами
- `movingObjectsFeatureCollection` / `movingObjectsGeoJson` — сборка GeoJSON:
  порядок координат `[lon,lat]`, ровно фиксированный набор свойств
  (`key,kind,label,heading,selected,stuck` — **и ничего про fleet/identity**),
  `heading==null → 0`, валидный round-trip JSON.
- `spiderfyCoincident` — раздвижка совпадающих объектов (поведение F4), константный
  экранный размер (метры от зума), некопланарные не трогаются.
- `MovingObjectKind.fromVehicleType` и наличие зарезервированных типов.

## Что переехало из донора `feature/timed-trajectory`

timed-математика A′ забрана **целиком**; поменялся только приёмник позиции.

- **Клиент:** `core/timed_trajectory.dart` (плеер плана: только вперёд, без
  телепорта, кламп к концу — как есть), интеграция в `vehicle_track_animator.dart`
  (`advanceTimed`, `TimedTrajectory` супер­седит from/to-ease при наличии плана+
  геометрии+as_of, иначе — прежнее консервативное поведение), модели
  `trajectory_point.dart`, поля `trajectory`/`asOf` в `AreaVehicle`/`Arrival`.
  17 донорских тестов плеера/аниматора прошли без изменений.
- **Бэкенд:** `parseTrajectory` (`transitProvider.ts`) строит план вперёд из
  `all_stations` (`second_left_by_route`), DTO аддитивно (`trajectory`+`as_of`),
  гейт флагом `timed_trajectory`. 30s-кэш/SWR/fan-out/upstream **не тронуты**.
- **Инструмент замера:** `core/fps_overlay.dart` — экранный счётчик fps, за
  `?fps=1` на staging-превью (на проде не включить). Пассивный (через
  `FrameTiming`), в покое кадры не гонит.
- **Аффинную проекцию** (`screen_projection.dart` / `ProjectedMarkerLayer`) из
  донора **не тащил** — карта сама проецирует символы. Знаю, что она там есть.

## Флаги и безопасность внедрения

- Два ортогональных флага (оба OFF prod / ON staging, KV, env-aware):
  - `symbol_layer` — **клиентский флаг рендера**: символьный GPU-слой vs виджеты.
  - `timed_trajectory` — **флаг данных/движения**: бэкенд отдаёт план вперёд,
    аниматор проигрывает его непрерывно.
  Работают все четыре комбинации; на staging включены оба.
- **Старый виджетный путь не удалён.** При `symbol_layer=OFF` строится прежний
  `WidgetLayer` c `VehicleMarker` — прод не рискует. Пути живут параллельно,
  переключаются флагом; одновременно никогда не рендерятся.
- **Деградация без краха:** не собрался источник/слой → `try/catch` оставляет
  `_vehLayerAdded=false` (позже ретрай), карта просто без символов ТС.
  Нет геометрии/плана → тихий фолбэк timed→консервативный ease.
- **idle=0 сохранён:** драйвер слоя (per-vsync `Ticker` в timed-режиме / 66 мс
  сэмплер в консервативном) сам останавливается, когда движения нет; в фоне —
  пауза (термо-фикс не нарушен). При простое источник не переписывается впустую.
- **Тап:** `queryRenderedFeatures`/`featuresInRect` по badge-слою → та же
  bottom-sheet линии. **Подсветка выбранного** — свойство `selected` на фиче
  (data-driven кольцо), **спайдерфай** — раздвижка координат в источнике. F4
  сохранено.

## Замеры fps (до/после, при разном числе объектов)

**«До» (виджетный путь, замеры донора — линейная стоимость):**

| ТС в кадре | fps (виджеты) |
|---:|---|
| мало | ~60 |
| ~40 (потолок fan-out) | ~19 |

Причина линейности зафиксирована в донорском постмортеме: на кадр — виджет-стоимость
+ `map.project` через JS-мост **на каждый** ТС. Это и есть узкое место, ради которого
задача существует.

**«После» (символьный слой) — сублинейно by design.** На кадр: одна
сериализация небольшого FeatureCollection (~40 фич ≈ 5 КБ) + один вызов
`updateGeoJsonSource` через мост + батч-рендер на GPU. 40 или 400 символов — та же
структура стоимости (растёт лишь размер строки, не число мостовых вызовов и не число
виджетов). Так делают зрелые карты движущихся объектов (Flightradar24, MarineTraffic,
Яндекс).

**Числовой device-fps — за владельцем через оверлей.** Причина та же, что у донора и
термо-фикса: достоверный fps знает только реальный телефон (iOS = WebKit, слабее чип,
DPR 3), а рендер maplibre-gl-js в моей среде не поднимается (встроенный браузер не
запускает WebGL-worker maplibre — та же граница, из-за которой и Flutter-карта, и
demo-стиль остаются пустыми). Поэтому: **сравнить symbol vs widget при 10 / 40 ТС на
живом iPhone через `?fps=1`** — см. «Как проверить». Порог приёмки: при много ТС в
центре символьный слой держит fps заметно выше виджетного и близко к 60, в покое — 0
кадров.

Искусственные 200+ ТС в кадре сегодня не воспроизвести (fan-out ≤12 остановок ≈ ≤40
машин); сублинейность проверяется архитектурно + станет наблюдаемой с городским API.

## Постфикс: зигзаг направленческой стрелки (после device-приёмки перфа)

Владелец подтвердил перф на телефоне (**3.5 мс/кадр** против 19–26 мс у
виджетов) и заметил зигзаг движения на **правильной** улице. Разбор:
- Позиция «монеты» — гладкая и следует кривой дороги: пишется из
  `RoutePath.pointAt(dispDist)` по **плотной** GTFS-геометрии (проверено: сегменты
  ~15–18 м, медиана 14–16 м). Привязка к геометрии при переносе в GeoJSON-источник
  **не потеряна** (проверено по коду + числовым тестом).
- Причина зигзага — **дискретный курс**: `headingAt` брал бэаринг одного сегмента
  при `dispDist`, т.е. новый курс каждые ~15 м, скачком 15–25° на изгибе. Стрелка
  вынесена на 34 px вперёд от монеты, поэтому её «нос» на каждом шаге прыгал вбок
  вдоль дороги.
- **Фикс:** непрерывный курс `RoutePath.headingAtSmoothed` — бэаринг упреждающей
  хорды (~30 м вдоль полилинии); оба конца скользят непрерывно с `dispDist`, курс
  поворачивает плавно сквозь изгиб. Применён в `TimedTrajectory.heading` и в
  `VehicleTrackAnimator.headingAt`. Юнит-тесты на непрерывность (`route_path_test`).

## Постфикс 2: набор объектов источника, spiderfy, трамвайная сеть

После device-приёмки перфа/зигзага владелец нашёл три регрессии символьного пути:

1. **ТС «встаёт намертво» / едет только то, что уже ехало / прыгает при подгрузке
   зоны.** Причина: контроллер плавности `_vehAnim.forward` запускался только при
   `!timedOn`. В timed-режиме ТС, чья геометрия/план ещё не подгружены (shape
   грузится лениво, или нет trajectory), падают в консервативный fallback-ease —
   а он без t-рампы контроллера не двигался: объект стоял на последнем фиксе и
   прыгал при следующем рефетче. **Фикс:** контроллер плавности запускается
   всегда при наличии движения; timed-плееры по-прежнему двигает vsync-тикер, оба
   пути сосуществуют. Плюс набор пишется в источник **сразу после каждого sync**
   (символьный источник write-driven — не «самолечится» на каждый rebuild, как
   виджет-слой), чтобы новые ТС появлялись, а ушедшие исчезали в тот же кадр.
   idle=0 сохранён (контроллер стартует только при реальном движении).
2. **spiderfy дёргался** при цикличном сближении/расхождении двух ТС (наезд
   ячеек). **Фикс:** стабильный spiderfy (`_stableSpiderfy`) — «сгруженность»
   удерживается ~0.5 с после расхождения (дебаунс), смещение **плавно
   затухает/нарастает** во время движения и **снапится на полную раздвижку** при
   разовой записи в покое (стоящая пачка раздвигается сразу, без застревания на
   25 %). Угол в группе — по стабильному рангу ключа. Frame-to-frame дёрганье
   убрано; idle=0 не нарушен (тикер ради веера не держится).
3. **Пропала трамвайная сеть.** Смешение императивных слоёв ТС с индексным
   менеджером декларативных слоёв пакета (`maplibre-layer-$index`) роняло
   декларативный `PolylineLayer` путей. **Фикс:** в symbol-режиме трамвайные пути
   — **свой императивный `LineStyleLayer` под слоями ТС** (`tram-rails`),
   которым владеет экран; виджет-путь (прод, флаг OFF) рисует их декларативно
   как раньше — не тронут.

> Баг сшивки ТС с чужим маршрутом («трамваи на дорогах без путей / стрелки в
> разные стороны») — **не этот слой**, отдельная ветка `fix/vehicle-route-mismatch`
> (данные/бэкенд). Символьный слой честно рисует, что дал аниматор.

## Постфикс 3: «стоячий синдром» и spiderfy-сквозное (смена проектных решений)

Владелец сменил прежние решения (зафиксировано):

- **Консервативный аниматор больше не даёт «встал намертво».** Причина синдрома:
  ТС с планом, но без загруженного GTFS-shape, падал в консервативный ease и
  стоял на фиксе до следующего рефетча. **Фикс — path-free экстраполяция:** если
  road-shape ещё нет, timed-плеер ведёт машину **вдоль точек самого плана**
  (станции по `second_left_by_route`), т.е. она сразу ЕДЕТ по предсказанию;
  когда shape подгружается — плеер **ре-анкорится** на него без прыжка
  (`TimedTrajectory.updatePlan` при смене геометрии проецирует текущую точку на
  новую полилинию). Донорская математика переиспользована целиком (плану просто
  подаётся геометрия из его же точек, `RoutePath.fromLatLon(plan)`), **не
  дублирована**. Консервативный ease остаётся крайним фолбэком (нет плана вообще)
  и путём прод-виджета за флагом.
- **Устаревание — гасим, не держим стоящей.** Прозрачность ТС = `opacityFor`
  (grace): пропавший из fan-out ТС плавно гаснет и убирается, а не висит стоящим.
  Проведено свойством `opacity` фичи → `circle/icon/text-opacity` выражения.
- **Fan-out расширен 12→18 остановок / 1000→1500 м** (`vehicles.ts`
  `MAX_STOPS_FANOUT`, клиент `_vehiclesMaxRadius`) — меньше «мёртвых» участков
  без свежих фиксов при листании. Каждый стоп по-прежнему на общем 30s-SWR;
  рост апстрим-нагрузки ≤~50% worst-case, 30s-кэш/SWR не тронуты. **Внимание:
  изменение fan-out НЕ за флагом** — на merge затронет и прод (это данные, не
  рендер; соответствует решению владельца). Staging меряет нагрузку первым; если
  источник упрётся — откат/гейт (STOP&ASK не потребовался при реализации).
- **Spiderfy переделан под сквозное прохождение (`_arrangeVehicles`):**
  - **раздвигаются только НЕПОДВИЖНЫЕ** совпадающие ТС (несколько на остановке/
    конечной читаются как несколько) — их позиции постоянны, веер стабилен без
    дебаунса; проезжающая мимо машина не схлопывает веер (группируем только
    стоящих);
  - **движущиеся НЕ раздвигаются** — при наезде **проходят друг сквозь друга**,
    накрывающий чуть уходит в прозрачность (crossing-dim 0.7, сглажено), затем
    разъезжаются по своим траекториям. **Прыжков нет.**

## Постфикс 4: расписательные объекты (scheduled) — карта, за флагом

Новый **источник/состояние** объекта (не новый механизм): scheduled-ТС — тот же
типизированный движущийся объект, что живой, но `source: "scheduled"`,
рисуется **полупрозрачным** и ведётся **той же timed-математикой** (план из
`stop_times` по геометрии). Реализована **карта** (моя часть — «рендер и
движение»); контракт согласуется с backend-сессией `gtfs-freshness`.

- **Модель:** `VehicleSource {live, scheduled}` + `tripId` на `AreaVehicle`
  (tolerant: всё, кроме явного `"scheduled"`, = live; поля опциональны — старый
  backend не ломается). Ключ scheduled = `sched:<trip_id>` — не коллизит с live.
- **Аниматор:** `source` проведён через `VehicleSample`/`VehicleTrack`; движение
  scheduled идентично живому (переиспользована timed-экстраполяция, план+as_of
  скармливаются независимо от флага `timed_trajectory`).
- **Рендер:** scheduled — базовая прозрачность `kScheduledBaseOpacity=0.5`
  (× grace/crossing), z-order **под** live (живой сверху при наложении). Свойство
  `opacity` символьного слоя уже поддержано.
- **Тап:** та же шторка линии + честная пометка «по расписанию — не живая
  позиция» (`vehicleScheduled`, EN/RU/SR).
- **Флаг `schedule_fallback`** (backend FEATURE_FLAGS + клиент, OFF prod): при
  OFF scheduled-объекты отбрасываются до аниматора (не рисуются). idle=0 сохранён
  (scheduled стоящий по плану = ноль кадров, как любой стоящий).
- **Дедуп** рейса live↔scheduled — за backend; клиент подстрахован префиксом
  ключа. Контракт: [`docs/SCHEDULE_FALLBACK_CONTRACT.md`](../SCHEDULE_FALLBACK_CONTRACT.md).

**Границы этой сессии:** сделана **карта**. **Гибридный список прибытий** (item 3
постановки) пересекается с работой сессии `gtfs-freshness` над экраном остановки
(GTFS∪live) — делается скоординированно там, на том же контракте (не наступаю на
чужую сессию). **E2E-проверка scheduled невозможна**, пока backend
(`gtfs-freshness`) не начнёт эмитить `source:"scheduled"`: проверено юнит-тестами
(классификация/движение/дедуп-ключ) + галереей; на staging заработает, когда обе
стороны включат флаг и backend отдаст объекты.

## Постфикс 5: merge main + фиксы после device-замера

**Подтянут `main`** (25 коммитов): фикс геометрии направления
(`vehicle_direction_shape` — ститчинг ТС по реальному направлению из
backend-resolved `route_id`), пригород, schedule Phase 1 (список). Конфликты
разрешены сохранением обеих сторон; клиент 155 тестов + backend 83 зелёные,
analyze чист.

- **«Едут сквозь дома» — половина уходит с merge.** Символьный путь теперь
  ститчит ТС по направлению: `shapeKeyOf(v) = byDirection ? v.routeId : v.line`
  кормит и `_ensureShapesFor(byRouteId:)`, и `path` семпла. При флаге
  `vehicle_direction_shape` (ON staging) геометрия — по направлению, не по
  канону. (Остаток сшивки — отдельная ветка `fix/vehicle-route-mismatch`.)
- **Дёргания на дальнем зуме — убраны.** Аффинной проекции в ветке нет (её не
  портировал); дёрганье шло от аранжировки (spiderfy/crossing по грубым в метрах
  ячейкам на overview → churn членства при движении камеры). Теперь **ниже
  detail-зума аранжировка выключена** — просто точки в истинных позициях
  (перекрытие точек не мешает), состояние сброшено.
- **Деградация в точки (F5) возвращена.** `circle-radius` даёт **маленькую
  точку** до detail-зума (≤6 px) и скачок в «монету» (12 px) на 15.5, где
  включаются символы глифа/номера/стрелки. На overview — точки, не стена монет.
- **Экстраполяция сдержана (не убегает).** `TimedTrajectory` предсказывает
  максимум **60 с / 500 м** вперёд от последнего фикса, потом держит; при свежих
  30-с фиксах кап не срабатывает (сбрасывается каждым фиксом), а при обрыве
  данных ТС ведёт ~минуту и останавливается, не «убегая по карте». Плюс **быстрее
  гашение** (grace-порог 4→3, круче кривая opacity), чтобы устаревший без данных
  ТС гас, а не висел. Оба кcapа — константы, легко подстроить по замеру.

## Постфикс 6: «пролетают/дёргаются» — инструментирован и пофикшен

Вместо догадок **инструментировал одну машину** живыми данными staging
(`scratchpad/capture_vehicle.py`): P93673 (линия 31), опрос 2.5 мин. Находка в
цифрах: **`as_of` заморожен** на 06:24:37 весь замер, GPS не двигался
(dGPS=0), но план твердит «653 м / 164 с». Причина — upstream подвисает (503),
SWR отдаёт замороженную доску, `now − as_of` растёт неограниченно.

**Корень «появляется далеко и летит»:** клиент ставил/вёл маркер на
`_distAtElapsed(now − as_of)` — при устаревшем `as_of` это `elapsed × темп(≈4 м/с)`
метров впереди реального (замороженного) GPS: 30 с → 120 м, кап 60 с → 240 м. Маркер
летел вперёд по плану, пока машина стоит.

**Фикс (принят владельцем), в `TimedTrajectory`:**
- **Появление и ведение — на реальном GPS** (точка плана 0), НЕ на
  `elapsed`-позиции. `build` ставит `wps.first.dist`; вход в кадр = на GPS-точке,
  без влёта-догона.
- **Короткий мостик `_bridgeSeconds=15`** вперёд от фикса (сглаживание до
  следующего), а не весь `now − as_of`.
- **Гейт устаревания `_stalenessSeconds=45`:** `now − as_of > 45 с` (доска не
  рефрешится) → не предсказывать, держать на фиксе, гасить по grace.
- Кап `_maxAheadMeters=500` — предохранитель. Всё — константы, подстраиваются.
- Тесты timed переписаны под модель «мостик от фикса + гейт устаревания»
  (появление на GPS даже при устаревшем `as_of`, короткий лид, стоп при
  устаревании, прогресс через рефреш, без отката).

**Замер 503 (по просьбе владельца — не fan-out ли давит источник):** опрос
`/vehicles/nearby` при `radius=400` (мало остановок) и `radius=1500` (18) —
**оба 100% 503** (14/14). Значит **503 = падение upstream, НЕ давление
fan-out** (узкий охват 503-ит так же). Латентность wide лишь +60 мс. **Радиус
откатывать не нужно.** Оговорка: пока upstream лежит, e2e-визуал не снять —
клиент показывает последнее и гасит по grace (ровно фикс); переверить, когда
источник поднимется.

## Известные компромиссы

- **Тип на монете = цвет + глиф.** Как у компакт-точки и как отмечено в промпте
  («цвета из `map_support`»). Три сигнала (номер + тип + стрелка) на месте.
- **Grace-fade появления/исчезновения** (плавная прозрачность в grace-period) в
  символьном режиме опущен — объекты появляются/исчезают резче, чем в виджетном.
  Мелкая деталь; легко добавить свойством `opacity` + выражением, отложено до
  device-приёмки визуала.
- **Аварийный люк** (богатый виджет для одного выбранного ТС поверх слоя) —
  опционален, в этой задаче не делал; масса — символы, выбранный подсвечивается
  на слое кольцом.
- **live_vehicles_map (мини-карта у остановки)** оставлена на виджетном пути —
  это «мало объектов», где масштаб не проблема; символьный слой нацелен на
  главную карту (сценарий «много ТС»).

### Замер fps на реальном iPhone (важно — за тобой)

Числовой fps на телефоне из моей среды не снять (WebKit, слабее чип, DPR 3;
maplibre-gl там тяжелее). Терминал не нужен:

1. Открой на iPhone (Safari): **https://preview-symbol.stigla.pages.dev/?fps=1**
   (пароль-гейт — из пасс-менеджера). В левом нижнем углу появится счётчик
   `NN fps · N.N ms` (зелёный ≥50, жёлтый 30–50, красный <30). Выключить — URL
   без `?fps=1`.
2. **Сравни при разном числе ТС:** зум района с 10 машинами vs центр с 40 —
   следи, как держится fps. Ключевой признак успеха: в центре (много ТС)
   символьный слой держит fps **заметно выше**, чем виджетный на проде при том же
   числе, и близко к 60.
3. Поводи/позумь во время движения ТС; заодно — **нагрев** телефона за пару минут.
4. **Порог приёмки:** при много ТС fps устойчиво выше виджетного пути и близко к
   60 без длительных просадок; в покое кадры замирают; тап открывает шторку;
   спайдерфай/подсветка работают. Если так — можно двигать к приёмке (напиши
   команду). Если fps низкий / телефон греется — скажи, что видел, вернёмся к
   оптимизации (первый рычаг — троттлинг записи в источник с vsync до ~30 Гц).
