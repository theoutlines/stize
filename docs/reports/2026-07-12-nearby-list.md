# Отчёт: экран «Рядом» — nearby-список в духе Transit

Дата: 2026-07-12 · Ветка: `feature/nearby-list` · Флаг: `nearby_list`
(OFF прод / ON staging) · Промпт: `docs/prompts/prompt_nearby_list.md`

## Коротко

Собран за фиче-флагом прототип nearby-списка: «я вышел на улицу — что мне
светит». Не редизайн — карта остаётся главным экраном; nearby живёт как
выдвижная панель (bottom sheet) поверх карты. Backend отдаёт готовые группы
«линия + направление» с ETA у ближайшей остановки; клиент только рисует.

Всё зелёное: backend `npx tsc --noEmit` чист, 53 vitest-теста; app
`flutter analyze lib test` без новых замечаний, 78 flutter-тестов. Задеплоено на
staging (backend + веб-превью), проверено живьём.

## Что сделано

### Backend — новый агрегирующий эндпоинт

`GET /api/v1/arrivals/nearby?lat=&lon=&radius=` (`src/lib/nearbyArrivals.ts`):

- **Тот же ограниченный fan-out, что у `/vehicles/nearby`**: ≤12 ближайших
  остановок, радиус ≤1500 м, каждый per-stop запрос едет на общем 30-секундном
  SWR-кеше. Лимиты не поднимал, кеш не обходил.
- **Группировка «линия + направление»**, дедупликация каждой группы до
  ближайшей к пользователю остановки (остановки идут по возрастанию дистанции —
  первая, обслуживающая группу, побеждает), сортировка строк по ближайшему ETA
  (тай-брейк по дистанции). Логика вынесена в чистую функцию
  `groupNearbyArrivals` и покрыта юнит-тестами.
- **Направление = конечная.** У прибытия конечной раньше не было. Беру терминал
  из геометрии рейса ТС (`all_stations`, последняя станция — это конец рейса,
  origin→destination) и резолвлю его в имя ближайшей GTFS-остановки
  (+ best-effort `direction_id`, если имя совпало с одним из двух GTFS-
  направлений линии). В `ArrivalDto` добавлены nullable `destination` /
  `direction_id` — это заодно обогащает и обычную карточку остановки на будущее.
- Явный Hono-роут (CORS покрыт общим `app.use("*")`); при killswitch отдаёт
  пустые группы, ошибки — `service_status: "unavailable"`.
- Флаг `nearby_list` добавлен в `FEATURE_FLAGS` (дефолт: OFF прод / ON staging,
  как все in-dev флаги — без записи в KV).

### Клиент — выдвижная панель над картой

- `NearbySheet` (`presentation/widgets/nearby_sheet.dart`): `DraggableScrollable
  Sheet` поверх карты (паттерн Google Maps/Transit). Свёрнута (peek ~30 %) —
  ручка + поиск + верх списка; вытянута (до 92 %) — полный список. Держит своё
  состояние: тянет данные от геопозиции пользователя, **автообновление раз в
  30 с** (зашито в `kLiveRefreshInterval`, не чаще — под кеш бэкенда) +
  pull-to-refresh. Состояния: нет геопозиции (подсказка + кнопка «Моё
  местоположение»), пустой радиус («Рядом остановок нет»), офлайн (кнопка
  «Повторить»), фильтр не нашёл.
- `NearbyList` (`presentation/widgets/nearby_list.dart`): **отдельный, чистый
  компонент** списка карточек (цветной бейдж линии по типу + `→ конечная` + имя
  ближайшей остановки и дистанция + ETA ближайших 1–2 отправлений), не завязан на
  sheet — принимает только `scrollController`/`onRefresh`, тестируется отдельно.
- Геопозиция **переиспользует существующий механизм карты** (`_meTo` из
  position-стрима, тот же `_recenterOnMe` для запроса доступа) — второго запроса
  пермишена не завёл. Тап по карточке открывает лист прибытий этой остановки на
  карте.
- Данные через существующий слой: `data/` (`NearbyArrivalsRepositoryImpl`) →
  `domain/` (`NearbyGroup`) → провайдер `nearbyArrivalsRepositoryProvider`;
  видимость — через `nearbyEnabledProvider` (`AsyncValue.valueOrNull`, не
  `.value`). Флаг выключен — sheet не монтируется, работает штатная поисковая
  строка карты.
- Локализация EN/RU/SR (все три `.arb` + `flutter gen-l10n`).

### Тесты

- Backend: `groupNearbyArrivals` (группировка / дедуп до ближайшей / сортировка /
  различение направлений по `direction_id`), парс терминала из `all_stations`,
  `nearestStop`/`getLineDirections` на реальном GTFS-бандле.
- Client: рендер `NearbyList` (бейдж/направление/ETA/дистанция) и состояния
  `NearbySheet` (нет геопозиции / список / офлайн / пусто), `kMapRenderingEnabled`
  не требовался — sheet без карты.

## Развилки [STOP & ASK]

1. **Бэкенд: новый эндпоинт vs переиспользовать `/vehicles/nearby`.**
   Выбран **новый `/api/v1/arrivals/nearby`**. Причина: `/vehicles/nearby` при
   дедупе по `garage_no` выбрасывает ETA и привязку к остановке — отдаёт только
   позиции машин. Для ETA-списка клиенту пришлось бы делать свой веер запросов по
   остановкам (лишний трафик, дубль серверной логики, обход шейпинга). Новый
   эндпоинт переиспользует тот же ограниченный fan-out и кеш — «правильное» место
   для агрегации. (Подтверждено владельцем.)

2. **Вход на экран: drawer / таб / что-то ещё.**
   Владелец выбрал третий вариант — **draggable bottom sheet над картой** (а не
   пункт drawer и не слот IndexedStack). Реализовано так; sheet за флагом, список
   вынесен отдельным компонентом, сохранёнки в списке не показываются (остаются в
   меню), сортировка v1 — по ETA. Жесты sheet↔карта не конфликтуют: панель висит
   в нижней части `Stack`, карта над ней панится штатно, перетаскивание —
   по области панели; отдельный STOP & ASK по жестам не понадобился.

3. **(Решено в рамках задачи, не выносилось.)** Источник «конечной». Данные
   прибытия конечную не несут; взял терминал `all_stations` → ближайшая
   GTFS-остановка. Альтернатива (матчинг геометрии ТС к двум GTFS-шейпам линии)
   тяжелее (чтение shape-ассетов на каждое прибытие) и не давала выигрыша для
   прототипа. Ограничение этого выбора — ниже.

## Сознательно отложено («дальше»)

- **Learning-сортировка** (по частоте использования линий) — v1 сортирует по
  ETA. Требует персистентной истории выбора; вне скоупа.
- **Пины избранного в списке** — сейчас избранное живёт в меню/«Моих
  остановках»; его влияние на ранжирование не трогаем.
- **Fleet-ID бейджи машин** — это отдельная фича (`feature/fleet-id`); строки
  без бейджей.
- **Универсальный поиск в sheet** — на surface с флагом ON вместо поисковой
  строки карты показывается nearby-панель, поиск в её шапке фильтрует **список
  локально** (линия/конечная/остановка), а не геокодит адреса. Полноценный
  универсальный поиск внутри sheet — на потом (при флаге OFF он на месте).

## Ложится ли формат на наши данные

**В целом — да, и убедительно.** На реальной геопозиции (Батутова, staging)
эндпоинт вернул 16 групп: `309 → Zvezdara /pijaca/` (ETA 2, 9), `46 → Mirijevo`
(3, 11), `79 → Dorćol` (6, 23), `7L → Ustanička` (6, 12), `55`, `14 → Ustanička`
(15, 37) и т. д. — ровно то «что мне светит», что и обещает формат: линия,
направление, ближайшая остановка с дистанцией, ближайшие отправления,
отсортировано по времени. ETA и дедуп до ближайшей остановки выглядят корректно.

**Главное ограничение — направление.** Оно берётся из `all_stations` прибытия, а
этот массив у части ТС в фиде отсутствует (та же причина, по которой у части
машин на карте нет heading). Тогда `destination = null` и строка рисуется как
`→ None` (в UI — фолбэк на номер линии). На тесте так вышло у трамваев `6`, `14`
и у части `7L`. Побочный эффект: одна линия может показаться дважды — одним
направлением с именем и «безымянным» хвостом. Это не ломает список, но снижает
доверие. Варианты на будущее: (а) прятать безымянную группу, если та же линия у
той же остановки уже показана с конечной; (б) для трамваев/троллейбусов, где
`all_stations` систематически пуст, резолвить направление матчингом позиции ТС к
двум GTFS-шейпам линии. Для прототипа фолбэк на номер приемлем.

Вывод: формат Transit хорошо ложится на автобусную часть данных; для
рельсового транспорта нужна доработка резолва направления. Идея живая — стоит
продолжать.

## Как проверить (владельцу, без терминала)

Всё уже задеплоено — нужен только браузер.

1. Открой веб-превью: **https://feature-nearby-list.stigla.pages.dev**
   (это persistent-превью ветки; продакшн `stigla.theoutlines.xyz` не затронут).
2. Появится запрос **Basic Auth** — логин/пароль в командном менеджере паролей
   (тот же, что для всех `*.stigla.pages.dev`, см. `docs/staging.md`).
3. Разреши браузеру доступ к геолокации, когда он спросит. Если не спросит —
   нажми на карте кнопку «моё местоположение» (прицел справа сверху): она
   запросит доступ и подтянет позицию.
4. Внизу поверх карты — **панель «Рядом»**. Что проверить:
   - в свёрнутом виде видно поле поиска и первые карточки;
   - **потяни панель вверх** — раскроется полный список линий вокруг;
     карточка = цветной бейдж линии (тип), `→ конечная`, ближайшая остановка +
     метры, ETA ближайших 1–2 отправлений; список отсортирован по времени;
   - **потяни список вниз** (pull-to-refresh) — данные обновятся; сами по себе
     они освежаются раз в 30 с;
   - введи номер/название в поиск — список отфильтруется;
   - тап по карточке — откроется лист прибытий этой остановки на карте.
   - Ожидаемо у некоторых линий (чаще трамваи) вместо конечной — номер линии:
     это известное ограничение данных, описано выше.
5. Состояния: выключи доступ к геолокации в браузере и обнови — панель покажет
   подсказку «включить геолокацию» с кнопкой (без страшной ошибки).
6. Что фича спрятана на проде: на `stigla.theoutlines.xyz` панели «Рядом» нет —
   там `nearby_list=OFF`, показывается штатная поисковая строка карты.

> **Важно про staging.** URL превью постоянный, но nearby-данные берутся с
> общего staging-воркера `stigla-api-staging.theoutlines.xyz`, а он один на всех
> и деплоится по требованию из любой ветки (см. `docs/staging.md`). Если другую
> ветку задеплоят на staging после меня, панель «Рядом» покажет офлайн/пусто
> (эндпоинта там не будет). В этом случае — просто попроси перевыкатить эту
> ветку на staging, это одна команда. На момент сдачи ветка выкачена и
> проверена: `/config` отдаёт `nearby_list:true`, `/arrivals/nearby` — HTTP 200
> с живыми группами.

Слияние в `main` — по твоей явной команде.
