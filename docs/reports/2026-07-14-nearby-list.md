# Экран «Рядом» — интеграция с расписанием-фолбэком (2026-07-14)

Ветка: `feature/nearby-list` · Задача: `docs/prompts/prompt_nearby_list.md`
(вариант B — не механический merge, а переосмысление поверх сегодняшней
инфраструктуры main). Флаги: `nearby_list` + `nearby_sort_board` (OFF прод / ON
staging).

## Контекст

«Рядом» (GPS → все линии вокруг без ввода, nearby-first как у Transit) писался
**до** массива фиксов 2026-07-14 и отстал на 54 коммита. За это время main
переписал ровно ту инфраструктуру, на которой стоит «Рядом»: модель направления
(`direction_route_id`/`trajectory` вместо `terminus`/`destination`) и
расписание-фолбэк (`getArrivals({includeSchedule})`, fan-out по остановкам).
Решение владельца: «Рядом» должна **опираться** на это, а не тащить свою старую
реализацию — иначе покажет пустые остановки там, где нет живого (та самая
пустота, что чинили весь день).

## Что сделано

### 1. Merge main, backend-конфликты — осознанно в пользу main
Из 11 конфликтов backend-часть (`arrivals.ts`, `transitProvider.ts`, `types.ts`,
тесты) разрешена в пользу main: сняты веточные `terminus` / `destination` /
`direction_id` / `resolveDirection`. Основа — сегодняшняя прод-модель
(`direction_route_id`, `trajectory`, расписание, кэш no-store, 503-фикс,
символьный слой), ветка садится поверх. Остальные конфликты (флаг-листы, l10n,
BACKLOG) — аддитивные, сохранены обе стороны.

**Бонус:** старый баг «→ None» (у части ТС пустой `all_stations` → нет
направления) ушёл почти полностью — направление берётся из надёжного
`direction_route_id` main, а имя конечной резолвится из GTFS-метаданных
(`getLineDtoByRouteId`, in-memory, без subrequests). На плотнейшей точке — 1
строка из 26 без имени (вариант линии `41A`, чей `route_id` не имеет конечной в
метаданных); edge-данные, не блокер.

### 2. Эндпоинт «Рядом» переписан на общую инфраструктуру
`getNearbyArrivals` теперь фанаутит по общему `nearbyStops` и на каждую остановку
зовёт `getArrivals({includeSchedule})` — **тот же путь, что список прибытий
остановки**, никакого своего fan-out/грид-индекса (удалены). Остановки рядом
несут **живое + плановое** (дедуп живое-первым уже внутри `getArrivals`), то есть
не пустуют ночью/в межпик. Группировка — по `line + direction_route_id`, дедуп до
ближайшей остановки, сортировка ETA / время-до-посадки (флаг `nearby_sort_board`).

### 3. Лимиты воркера — замерено, не угадано
`getStopSchedule` — 1 **некешируемый** subrequest на остановку; 18-остановочный
map-путь с расписанием утром пробил CPU/subrequests (503). Поэтому расписание
наследуется только для **ближайших N** остановок fan-out (остальные — live-only):
ближние — ровно те, куда дойдёшь пешком, там «не пусто» и важнее.

Замер на staging в **плотнейшей точке Белграда** (Trg Republike, 30 остановок в
500 м; `wrangler tail`):

| `schedule_stops` | HTTP | cpuTime | scheduled-строк | outcome |
|---|---|---|---|---|
| 0 (live-only) | 200 | 16–29 ms | 0 | ok |
| 3 | 200 | 21 ms | 18 | ok |
| **5 (дефолт)** | 200 | **26 ms** | 22 | ok |
| 8 (весь fan-out) | 200 | 53 ms | 26 | ok |

503 не воспроизвёлся в достижимом диапазоне (nearby капит fan-out на 8; фатал был
на 18). Дефолт — **K=5** (26 мс, половина ~50-мс бюджета воркера, запас на пик:
замер сделан вне часа пик). K **настраивается в рантайме** через KV
`config:nearby_schedule_stops` (clamp 0..8) — поднять/опустить без пересборки,
как флаг, если дальние остановки будут пустовать.

### 4. Экран (клиент)
`NearbySheet` подшит в переписанный main-`home_map_screen` за флагом `nearby_list`
(замещает нижнюю поисковую строку). Привязка — к позиции пользователя (`_meTo`,
из сегодняшнего position-стрима, не к камере карты), polling 30 с (кэш backend),
pull-to-refresh, поиск. Тап по карточке открывает лист прибытий этой остановки
(тот же оверлей, что тап по пину). Плановое ближайшее отправление помечено
существующим ярлыком «По расписанию» — видно, что это never-empty хвост, а не
живой ТС. Сегодняшняя работа экрана (императивные избранные, символьный слой,
FPS-overlay) не тронута.

## Известные ограничения / отложено
- 1 строка из 26 «→ None» на плотнейшей точке (вариант `41A` без конечной в
  GTFS-метаданных) — редкий data-edge.
- `nearby_sort_board` использует константный штраф вместо реального headway (как и раньше).
- Отложено (как и в первой версии): learning-сортировка, пины избранного, Fleet-бейджи в карточке.

## Как проверить (владельцу, без терминала)

Всё уже задеплоено — нужен только браузер.

1. Открой веб-превью: **https://feature-nearby-list.stigla.pages.dev**
   Появится **Basic Auth** — логин/пароль в командном менеджере паролей (те же,
   что для всех `*.stigla.pages.dev`, см. `docs/staging.md`). Вверху — амбер-бейдж
   **STAGING** (значит превью смотрит на staging-backend, флаги ON).
2. Разреши браузеру геолокацию, когда спросит. Если не спросил — нажми прицел
   справа сверху («моя позиция»): он запросит доступ и подтянет позицию.
3. Внизу поверх карты — панель **«Рядом»** (заменяет обычную строку поиска, т.к.
   флаг `nearby_list` ON на staging). Потяни вверх — полный список линий вокруг:
   бейдж линии (тип), `→ конечная`, ближайшая остановка + метры, ETA ближайших 1–2.
4. Главное, что проверяем (суть задачи — **остановки рядом не пустуют**):
   - в списке есть строки с пометкой **«По расписанию»** под временем — это
     плановые отправления там, где сейчас нет живого ТС (ночью/в межпик их
     больше). Именно они не дают панели быть пустой;
   - живые строки (с ETA без пометки) идут первыми;
   - тап по карточке — откроется лист прибытий этой остановки на карте;
   - поиск по номеру/названию фильтрует; pull-to-refresh обновляет (сам — раз в 30 с);
   - выключи геолокацию в браузере и обнови — панель покажет подсказку «включить
     геолокацию», без страшной ошибки.
5. Дизайн-слепок панели (для Figma): **https://preview-gallery.stigla.pages.dev**
   → страница «Nearby — list» (последняя карточка — со статусом «Scheduled»).

> **Про staging.** URL превью постоянный, но данные «Рядом» берутся с общего
> staging-воркера `stigla-api-staging.theoutlines.xyz` (один на все ветки,
> деплоится по требованию). На момент сдачи ветка выкачена и проверена: `/config`
> отдаёт `nearby_list:true` + `schedule_fallback:true`, `/arrivals/nearby` в
> центре — HTTP 200 с живыми и плановыми группами. Если другую ветку выкатят на
> staging после меня и панель опустеет — попроси перевыкатить nearby-list на
> staging (одна команда).
>
> **Тюнинг без пересборки.** Если в проде увидим, что дальние остановки часто
> пустуют, число остановок с расписанием поднимается KV-ключом
> `config:nearby_schedule_stops` (замер показал: до 8 влезает), без нового билда.

Слияние в прод — по твоей явной команде.
