# Диагностика: повторный алерт «50% дневного лимита KV-операций» (read-only)

**Дата:** 2026-07-22 · **Ветка:** `diag/kv-ops-spike` · **Тип:** диагностика, без
изменений кода/деплоев/флагов.

**Триггер:** письмо Cloudflare 2026-07-22 00:52 UTC (02:52 Белград) —
«[Alert] Approaching your KV daily operation limit — 50% reached». Это **второй**
KV-алерт (первый — 2026-07-21, см. `2026-07-21-sweep-state-d1.md`).

## TL;DR (вердикт)

**Горящая метрика — не reads, а writes.** Reads физически не могут дать «50%»:
их пик ~8.6k/сутки = **8.6 % от лимита 100k/день**. 50 % можно достичь только по
**writes** (лимит 1000/день, 50 % = 500).

Сейчас writes ≈ **0** (22 июля = 0; за трейлинг-24ч = 160 и затухают в ноль). В
момент алерта (00:52 UTC) и reads, и writes, и трафик воркера были практически на
нуле — **алерт не может быть real-time про 22 июля**. Это **отложенное
уведомление**, ссылающееся на всплеск writes ДО фикса — persistence состояния
sweep в KV, который **уже устранён** переносом в D1 (миграция 0007, отчёт
`2026-07-21-sweep-state-d1.md`).

**Категория: (a) регрессия — но уже исправленная.** Не органический рост reads,
не внешний скрапер. Новых действий не требуется; второй алерт — эхо первого
инцидента.

---

## 1. Метрика и таймлайн

Все числа — из дашборда Cloudflare, namespace **STIGLA_KV**, прод. Аккаунтные
идентификаторы и хост источника опущены намеренно (публичный репозиторий).

### Абсолютные счётчики

| Окно | Reads | Writes | Deletes | Lists |
|---|---|---|---|---|
| Месяц (1–22 июля, накопительно) | 153.58k | 790 | 20 | 10 |
| Трейлинг-24ч (≈21.07 08:00 → 22.07 08:00 UTC) | 8.6k | 160 | 10 | 0 |
| Текущий день 22.07 (00:00 → 06:48 UTC) | 1.32k | **0** | 0 | 0 |

Дополнительно: storage 25 kB / ~25 ключей; read-latency здоровая (P50 6 ms,
P99 51 ms); read-status за 24ч — hot 4.8k, cold 1.7k, not-found 2.1k
(«not-found» = `KV.get()` по неустановленным ключам, в основном дефолтные
флаги на проде — см. §2).

### Форма (график Operations, трейлинг-24ч)

- **Reads (синяя линия):** классическая **органическая дневная кривая** — пик
  ~1150 reads/час (~0.32/s) днём 21 июля (после 12:00 UTC = дневной час-пик в
  Белграде, двойной горб), резкий спад после 16:00 UTC и **дно ~0.02/s (~70/час)
  всю ночь**. К 08:00 22 июля — лёгкий подъём (утро).
- **Writes (оранжевая линия):** крошечный всплеск в самом начале окна (~08:00
  21 июля) и далее **ровный ноль** весь период. Это **затухающий хвост**, а не
  постоянный источник.

### Лимиты бесплатного тарифа (для привязки)

| Класс | Лимит/день | 50 % | Наблюдаемый пик/день | % лимита |
|---|---|---|---|---|
| Reads | 100 000 | 50 000 | ~8.6k | ~8.6 % |
| Writes | 1 000 | **500** | ~160 (24ч), 0 (22.07) | ≤16 % сейчас |
| Lists | 1 000 | 500 | 0–10 | ~0 % |
| Deletes | 1 000 | 500 | 10–20 | ~1–2 % |

### К какому UTC-дню относится алерт

**Не к 22 июля.** На 00:52 UTC (52-я минута суток) при ночном темпе ~0.02/s
накопилось бы ~60 reads и 0 writes — до 50 % любого лимита несопоставимо далеко.
Значит письмо **отложено** и относится к предыдущему периоду. Единственный класс,
способный пробить 50 % дневного лимита, — **writes (500/день)**, а единственный
известный источник массовых KV-writes — persistence состояния sweep **до** его
переноса в D1 (20–21 июля, до midday-фикса). Cloudflare-алерты по usage регулярно
запаздывают и суммируют завершившийся UTC-день — картина полностью сходится с
«эхом» первого инцидента.

## 2. Карта KV-операций по коду (`backend/src`)

Байндинг `STIGLA_KV`. Классификация по триггеру + оценка ops/день под наблюдаемым
трафиком (воркер: ~4k invocations/сутки).

### Per-request (живой трафик)

| Путь | KV-операции на запрос | Примечание |
|---|---|---|
| `GET /config` | **15 reads** (14 флагов `getAllFlags` + `donate_url`) | `featureFlags.ts:156`, `index.ts:69,75`. Клиент дёргает **1 раз за сессию** (`app_config` — `providers.dart:56`, без авто-refresh на resume). |
| `GET /arrivals?stop=` | 2 reads (`isServiceKilled` + `jam_detection_collect` memo); +1 (`upstream_budget`) на реальном SWR-refresh | `killswitch.ts:5`, `arrivals.ts:202`, meter — только на refresh, не на cache-hit |
| `GET /vehicles/nearby` | ~2 reads (`isServiceKilled` + `jam` memo на весь fan-out) | memo по `ctx` схлопывает fan-out ≤12 стопов в 1 чтение флага |
| `GET /arrivals/nearby` | ~3 reads (`isServiceKilled` + `nearby_sort_board` **не-memo** + `jam` memo) | `nearbyArrivals.ts:222` использует `getFlag` (не memoized) |
| `GET /jams` | — | на проде `jam_detection_show` OFF → клиент не вызывает |
| `GET /geocode` | 1 read + **1 write на miss** (TTL-кэш) | `geocode.ts:18,42`; объём низкий (поиск адреса) |
| `POST /ideas`, `/ideas/:id/comments` | 1 read + **1 write** (rate-limit ключ) | `ideas.ts:37,40` |
| `POST /feedback` | — на проде | `feedback_form` OFF → 403 до чтения/записи (`index.ts:491`) |
| `POST /events` | 1 read (`product_analytics` memo) | на проде OFF → 204, запись в D1, не KV |

**Множитель reads = `/config` (15×).** Кросс-запросного (per-isolate) кэша флагов
**нет** — есть только memo в пределах одного запроса (`getFlagMemoized`,
`featureFlags.ts:143`, `WeakMap` по `ctx`). То есть каждая новая сессия платит 15
KV-reads. Это осознанный трейд-офф ради instant-flip семантики, и при текущем
объёме reads (9 % лимита) он безвреден — но это и есть «история», если однажды
именно reads подойдут к лимиту.

### Per-cron-tick (1440/день, `* * * * *`)

Sweep-тик на **проде** читает **1 флаг** `analytics_sweep` (`sweep.ts:180`) и,
поскольку тот **OFF**, немедленно выходит (`return noop("disabled")`,
`sweep.ts:185`) — **до** чтения `upstream_budget`, budget/breaker-конфигов и
интервалов. Итого cron = **1 read/тик = 1440 reads/день (~1.4 % лимита)**,
**0 writes**. Новые ключи `config:upstream_budget_*` / `config:breaker_*` /
интервалы sweep (`upstreamBudget.ts:105–133`, `sweep.ts:249–252`) на проде
**не читаются вовсе**, пока `analytics_sweep` OFF.

> Именно так первый инцидент и «лечился»: раньше sweep писал состояние (курсор и
> т.п.) в KV каждый тик → до ~1440–2880 **writes/день**, что и пробивало лимит
> writes. После переноса состояния в D1 (миграция 0007) sweep-writes в KV = 0.

### Rare / manual (≪ лимитов)

Флип флага (`setFlag`, `featureFlags.ts:153`), killswitch (`killswitch.ts:11`),
дневной cron alerts (`alerts.ts:260`, 1–2 writes/день). Пренебрежимо.

### Итог оценки reads/день

`/config`: пусть ~300–400 сессий/день × 15 ≈ 4.5–6k reads.
Поллеры + одиночные `/arrivals` (по ~2–3 read): остаток трафика.
Cron: 1.44k. **Сумма ≈ 8–9k reads/день** — совпадает с наблюдаемыми **8.6k**.

## 3. Сверка (Step 3) и вердикт с арифметикой

**Reads.** Воркер ≈ **4k invocations/сутки** × **~2 KV-reads/invocation** ≈ **8k**
≈ наблюдаемые **8.6k reads/сутки**. Сходимость ~в пределах 10 % — как и требовалось
(уровень отчёта sweep-state). Кривая reads повторяет кривую запросов воркера
(органика), а не плоский ночной профиль скрапера.

**Внешний скрапер исключён.** График запросов воркера — активность днём 21 июля
(13:45–17:15 GMT+2) и **практически ноль ночью** (20:45 21.07 → ~08:00 22.07).
Окно 22:00–02:00 UTC (00:00–04:00 GMT+2), на которое пришёлся алерт, — пустое.
Cache-hit воркера 97 %, ошибки 72/24ч (4xx = 7) — фона аномалий нет. Субзапросы к
источнику ~6k/24ч — это штатные SWR-refresh + map-fan-out, не KV.

**Writes.** Текущие writes ≈ 0 (22.07) и 160 (24ч, затухающий хвост) — оба **ниже**
порога 500. День, реально пробивший 50 % writes, лежит **до** видимого окна (до
~08:00 21.07 / 20.07) — это pre-fix sweep-writes.

### Вердикт

**(a) Регрессия — уже исправленная.** Горела метрика **writes**; корень — KV-
persistence состояния sweep; фикс (перенос в D1, миграция 0007) **уже в проде** и
уронил writes до ~0. Алерт 22.07 00:52 UTC — **запоздалое/повторное уведомление**
о том же завершившемся всплеске writes, а не новый real-time пробой.

Это **не** органический рост reads (reads на 8.6 % лимита) и **не** внешний трафик
(ночь пустая).

## 4. Рекомендация

1. **Действий не требуется прямо сейчас.** Root-cause writes уже устранён;
   reads с большим запасом (9 % лимита). [STOP & ASK]-сценарий (пробой до 00:00
   UTC, 429 на живом трафике) **не наступает**.
2. **Постоянный фикс — уже на месте** (sweep-state → D1). Достаточно **подтвердить**
   на ближайших суточных границах, что writes остаются ~0 и новый «50 %»-алерт для
   дня с writes < 500 не приходит (см. §5). Если через 1–2 суток тихо — закрыть.
3. **Workers Paid ($5/мес, 10M reads / 1M writes) сейчас НЕ нужен.** Он лечит
   *симптом* (потолок), а *причина* (writes) уже вылечена в коде. Апгрейд имеет
   смысл только если органические **reads** поползут к ~50k/день (сейчас ~9k) —
   тогда $5 снимут потолок навсегда без изменений кода. Как разовая «страховка от
   эха» апгрейд тоже сработал бы, но это переплата за уже решённую проблему.
4. **Мелкая оптимизация reads — отдельной задачей, НЕ сейчас.** У чтения флагов/
   конфига нет per-isolate кэша: `getFlag`/`getAllFlags` (`featureFlags.ts:131,156`)
   ходят в KV на каждый запрос, поэтому `/config` = 15 reads/сессия. Если reads
   когда-нибудь приблизятся к лимиту — добавить короткий (напр. 30–60 s) isolate-
   level TTL-кэш поверх `getFlag` (осторожно с instant-flip семантикой флипов).
   Сегодня не требуется (reads 9 % лимита) — фиксируем как бэклог, не трогаем.

## 5. Как проверить (владельцу, без терминала)

Всё — в дашборде Cloudflare, read-only:

- **Storage & databases → Workers KV → STIGLA_KV → вкладка Metrics:**
  - **Summary** — крупно Reads / Writes / Lists / Deletes за месяц.
  - **Operations** — переключатель **Requests/second ↔ Request count** (справа
    сверху). В режиме *Request count* цифры «Read / Write / List / Delete» за
    последние 24ч. **Ключевой признак:** оранжевая линия Write у нуля → writes не
    горят; синяя Read — дневной горб + ночное дно → органика, не скрапер.
  - **Read status** — hot / cold / **not-found** (много not-found = чтения
    неустановленных дефолтных флагов, это норма).
- **Workers KV → список namespaces → блок Usage** (пикер даты сверху): Reads /
  Writes / Deletes / Lists **за текущий день**. Быстрый ответ «сколько уже нажгли
  сегодня».
- **Compute → Workers & Pages → stigla-backend → Metrics:** карточки
  **Invocations / Errors / Cache hit rate** и график **Asset requests by status**.
  Плоский ночной участок = нет ночного скрапера. Полезное эмпирическое правило:
  **KV-reads ≈ 2 × Invocations**.

**Правило чтения алерта:** если письмо «50 % reached» приходит вскоре после 00:00
UTC, а у дня-кандидата reads < 50k и writes < 500 — это **запоздалое** уведомление
о предыдущем дне, а не текущий пробой. Сверять по дневным счётчикам, а не по
времени письма.
